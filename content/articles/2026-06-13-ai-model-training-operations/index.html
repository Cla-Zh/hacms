一、核心发现摘要

发现一：训练基础设施已进入"AI Factory"时代——软硬协同设计成为新范式

NVIDIA DGX SuperPOD、Google TPU Pod、Meta 35,000+ H100集群、Tesla Dojo代表四种不同的AI Factory路线：通用超级计算（NVIDIA）、自研芯片+框架协同（Google TPU+Pathways+JAX）、开源极限扩展（Meta PyTorch）、领域定制晶圆级封装（Tesla Dojo）。核心趋势：芯片、互联、框架的联合优化（Co-design）比单点突破更能提升训练效率。

NVIDIA DGX SuperPOD扩展至数千GPU，提供"交钥匙"AI Factory方案，支持兆级参数模型训练。nvidia.com/en-us/data-center/dgx-superpod

发现二：经典模型与Transformer长期共存——混合架构是生产现实

Tesla Autopilot从HydraNet（CNN）演进到BEV+Transformer，但量产车型CNN backbone至今仍在使用；Google Translate从LSTM迁移到Transformer经历了三年渐进式混合架构；字节推荐系统Wide&Deep+实时更新仍在支撑日均数百亿次推荐请求（Meta官方表述）。洞察：不是"谁替代谁"，而是各取所长——CNN的推理效率、Transformer的表征能力、RNN的时序建模，在不同场景各司其职。

Tesla Autopilot FSD Chip上，BEV+Transformer推理延迟 <10ms（8相机输入），而量产车型仍广泛保留CNN backbone以平衡功耗与延迟。tesla.com/AI — Tesla AI Day

发现三：团队组织正在结构性重构——Forward Deployed Engineer成为AI原生公司标配

Scale AI首创的FDE（Forward Deployed Engineer）模式已被OpenAI、Anthropic、Cohere、Mistral等公司复制——工程师嵌入客户现场，从"写代码"转变为"写spec定义问题"。Anthropic以约5000人支撑$30B ARR，CEO仅1个直接下属，System Designer/Implementer 50/50分工是核心组织创新。洞察： intelligence-to-headcount比率已永久性改变，PM需要能交付简单功能。

Scale AI的FDE模式被OpenAI/Anthropic/Cohere/Mistral复制，Forward Deployed Engineer是2026年AI公司增长最快岗位。getperspective.ai — Scale AI Forward Deployed Engineers

发现四：MLOps全生命周期=持续训练+金丝雀+回滚——三环缺一不可

Netflix每晚全量重训练+每5分钟在线更新+金丝雀<30秒回滚，是生产ML系统的最佳实践模板。Meta推荐系统每15-60分钟增量更新Embedding层，每天全量重训练排序模型，增量更新后AUC与全量训练差异<0.5%。核心原则：Champion模型全程热备，回滚是流量配置变更而非重新部署。

Google MLOps三级成熟度模型：Level 0（手动）→ Level 1（持续训练CT）→ Level 2（CI/CD流水线自动化）。cloud.google.com — MLOps Continuous Delivery

发现五：推理架构从"单点优化"进化到"系统级协同"——vLLM/Paged Attention成LLM推理事实标准

TensorRT-LLM、ONNX Runtime、vLLM三分天下推理框架市场。vLLM的Paged Attention通过分页内存管理将LLM推理吞吐提升24倍，已被OpenAI、Anthropic、字节豆包广泛采用。KServe+Istio成为Kubernetes上模型推理的标准组合，支持灰度/A-B/Canary多种发布策略。

NVIDIA TensorRT-LLM相比CPU可提升最高36倍推理性能，支持FP8量化和张量并行。developer.nvidia.com — TensorRT

发现六：国产AI基础设施全栈崛起——昇腾/昆仑/PAI已具备规模生产能力

华为昇腾910B（对标H800）、Atlas 900 SuperPOD、MindSpore自动并行；百度昆仑芯3代（256 TFLOPS）、百舸异构计算平台；阿里PAI-EPL弹性并行；字节10,000+ H800/H100集群、火山引擎ML平台。核心能力：国产集群已从"可用"进入"好用"阶段，故障自动恢复、30天不中断训练已成国内厂商标配。

2.1.4 Tesla Dojo — 晶圆级定制路线

Tesla Dojo采用与NVIDIA/Google完全不同的领域定制路线：

  • D1 Chip：362 TFLOPS（FP16），354核定制CPU，片内带宽10TB/s，功耗约500W TDP
  • 晶圆级封装：每个Dojo Cabinet含12个节点（12×D1），算力4,344 TFLOPS（FP16），通过2D Mesh Torus网络互联，延迟<1微秒
  • 无DRAM设计：利用大规模SRAM片上存储（约440GB SRAM per WSE），消除外部显存带宽瓶颈
  • 训练收益：Tesla AI负责人Andrej Karpathy披露，Dojo单节点训练效率约为A100的4倍（特定workload）
  • 应用：训练FSD感知模型，处理每秒数百万帧视频数据；Dojo超算提供1.8 EFLOPS算力
Tesla Dojo D1芯片采用晶圆级封装，片内带宽10TB/s，是自动驾驶AI训练的专用超算。tesla.com/AI — Tesla AI Day

四、经典模型生产运维（CNN/RNN/LSTM）

4.1 Tesla Autopilot — CNN架构三次大迭代

第一代 HydraNet（2019）：多任务CNN

  • RegNetY-64GF backbone + 8个独立heads（目标检测/车道线/可行驶区域等）
  • 特征缓存机制：每帧特征提取结果复用，降低重复计算开销
  • 多任务联合训练，避免孤岛模型问题

第二代 BEV+Transformer（2021-2022）

  • 8相机构成360°视野，通过Transformer融合到统一3D BEV空间
  • ViT替代RegNet作为backbone；Occupancy Network替代2D检测头
  • FSD Chip推理：BEV+Transformer延迟 <10ms（8相机输入）
  • mAP相比2D检测提升约12%

第三代 端到端大模型（2023-2024）

  • 取消BEV直接输出规划，>1亿参数（vs HydraNet约5000万参数）
  • Dojo超算约1.1 EFLOPS算力（ExaPOD设计值）训练
  • 量产车型CNN backbone仍广泛存在——功耗/延迟/稳定性的最优解
Tesla Autopilot从HydraNet演进到BEV+Transformer，量产车型仍保留CNN backbone，混合架构是生产现实。arXiv:2003.10755 — HydraNet

4.2 Google Translate — LSTM→Transformer三年迁移

Google Translate从2016年LSTM迁移到Transformer，采用了三年渐进式混合架构策略：

  • Phase 1（2017-2018）：Transformer Encoder + LSTM Decoder混合架构
  • Phase 2（2018-2019）：全Transformer架构
  • 最终收益：BLEU 24.3→26.5（+9%），延迟降低47%
Google LSTM到Transformer渐进迁移，BLEU+9%，延迟-47%；保留了LSTM Decoder作为小语种翻译快速回滚方案。ai.googleblog.com — Transformer Translation

10.3 标准流程：6阶段闭环

7-10人团队的标准ML生产流程，每个阶段必须有明确交付物：

① 数据接入 → ② 实验调优 → ③ 训练上线 → ④ 灰度验证 → ⑤ 监控迭代 → ⑥ 自动重训练

① 数据接入（Data Engineer主导）

  • 数据源接入：Kafka/Flink实时流 或 S3/HDFS批量数据
  • 数据质量门禁：TFDV/Great Expectations，自动检查特征分布
  • DVC数据版本化：每个训练数据集打版本标签，存入MLflow元数据
  • 交付物：数据质量报告 + 数据版本号

② 实验调优（ML Engineer主导）

  • 基线模型：用最小数据集（1%样本）跑通全流程，确认Pipeline可行
  • 实验记录：每次训练必须记录在MLflow，附超参/数据版本/评估指标
  • 超参搜索：W&B Sweeps或Optuna，优先手动调参再自动化
  • 交付物：MLflow实验记录 + 候选模型列表 + 线下评估报告

③ 训练上线（ML Engineer + ML Ops协作）

  • Kubeflow Pipeline：数据验证→特征工程→训练→评估→模型注册，全自动
  • Model Registry：训练完成的模型自动注册到MLflow Model Registry（Staging阶段）
  • 模型优化：TensorRT/ONNX量化，延迟/吞吐满足SLO
  • 交付物：模型文件 + 训练日志 + 评估指标 + Serving镜像

④ 灰度验证（ML Ops主导）

  • Shadow Mode（24小时）：新模型并行接收流量但不响应，验证无错误
  • 0.1% Canary：Istio流量分割，监控p99延迟+错误率+业务指标
  • 观察期：每阶段至少4小时，Guardrail红线：p99<2x champion、Error Rate<0.5%
  • 全量上线：确认无异常后切100%流量，旧模型保留为Champion热备
  • 交付物：灰度报告 + 金丝雀指标截图 + 上线确认

⑤ 监控迭代（ML Ops + SRE主导）

  • 技术指标：Prometheus metrics（Grafana看板），AUC/Latency/Throughput/Error Rate
  • 漂移检测：Evidently AI，PSI>0.2自动告警Slack
  • 业务指标：CTR/Retention，每日Review，关键指标下跌>2%触发Review
  • 交付物：每日监控日报 + 每周模型健康度Review

⑥ 自动重训练（ML Ops + Data Engineer协作）

  • 触发条件：数据漂移（PSI>0.2）或定时（每日凌晨）或性能下跌
  • Pipeline自动执行：数据验证→增量训练→新模型上线灰度→通过后自动全量
  • 回滚机制：任何Guardrail异常，自动切回Champion，无需人工干预
  • 交付物：重训练日志 + 模型对比报告 + 自动上线确认

10.4 团队协作规范（必须成文的制度）

模型生命周期管理规范

  • 模型命名规范：{model_name}_{version}_{date}，如 recommender_v2.3_20260613
  • 模型阶段流转：Development → Staging → Production → Archived，每阶段有明确审批
  • 实验记录规范：每次训练必须附实验卡片（MLflow），含动机、方法、数据版本、结论
  • 模型下架规范：任何模型下线需数据支撑（Performance < baseline + 漂移确认）

代码与数据规范

  • 代码版本：Git，所有ML代码提交到Git，CI/CD触发训练Pipeline
  • 数据版本：DVC，数据集变更必须DVC commit并附变更说明
  • 模型Artifact：MLflow Model Registry，原始模型+量化模型分开注册
  • 特征定义：Feature Store，所有特征有文档、数据owner、质量指标

On-Call与故障响应

  • On-Call轮值：ML Ops Engineer + Platform Engineer双人On-Call，每周轮换
  • P0告警：p99 > 200ms 或 Error Rate > 1%，5分钟内响应，15分钟内定位
  • P1告警：AUC下跌 > 1% 或 漂移检测触发，1小时内响应
  • 故障复盘：任何P0/P1故障，48小时内必须输出复盘报告

10.6 总结：核心原则与避坑指南

7-10人团队MLOps建设的5条核心原则

  1. 先用开源跑通全流程：不要在第一个月就买商业平台，先用MLflow+DVC+Kubeflow跑通，证明价值后再按需采购
  2. ML Engineer必须参与Ops：不是"ML Engineer写模型，Ops上线"，两者必须共同编写Pipeline，职责边界要清晰但不能割裂
  3. 监控是生命线，从第一天就要有：不要等模型上线后再加监控，上第一个基线模型时就接入Prometheus，建立baseline
  4. Champion热备，回滚<30秒：任何新模型上线前，必须确认旧版本Champion在线且可立即接管流量
  5. 数据质量比模型调参更重要：花70%时间在数据质量门禁（DVC+TFDV），30%时间在模型调参，这是投入产出比最高的决定

常见5大坑（提前避免）

  • 坑1：买了商业ML平台但没人会用 → 先用开源验证团队能力，再按需采购
  • 坑2：模型上线后没有监控，等用户投诉 → 从第一个模型就开始监控，这是技术债，越早还越便宜
  • 坑3：实验结果不可复现 → DVC数据版本+MLflow实验记录，每次训练必须记录，数据和代码缺一不可
  • 坑4：GPU利用率低（<30%） → 用Kubeflow + Spot Instance，动态调度，不用时释放训练资源
  • 坑5：没有数据质量门禁，脏数据污染模型 → TFDV/Great Expectations做数据质量门禁，数据问题在训练前发现比训练后发现成本低100倍

11.1 传统模型 vs 大模型：开发范式有何不同？

传统ML模型（CNN/RNN/LSTM/DNN/Wide&Deep）与LLM在开发流程上有根本性差异：

维度传统ML模型（CNN/RNN/LSTM）大语言模型（LLM）
模型规模数万~数亿参数数十亿~万亿参数
训练数据结构化/标注数据，GB~TB级非结构化文本，TB~PB级
训练硬件单卡/单机多卡，8×A100即可万卡集群，InfiniBand网络
推理成本$0.001~0.1/千次$0.01~1/千tokens
迭代周期天~周级，数据驱动月~季度级，评估驱动
开源生态sklearn/PyTorch/Keras成熟HuggingFace/vLLM/DeepSpeed
典型场景推荐/风控/视觉/NLP分类对话/生成/代码/推理
传统ML模型并没有被LLM取代——推荐系统（Wide&Deep/DCN）、风控模型（GBDT+LR）、视觉感知（CNN+ViT混合）、语音识别（LSTM+CTC）仍是生产主流。Meta Engineering - Journey to 1000 Models

11.8 小结：传统模型开发的五条核心工具链

场景一：视觉分类/检测/分割（CNN）

  1. 数据增强：Albumentations
  2. 预训练backbone：timm 或 torchvision
  3. 训练框架：PyTorch Lightning
  4. 模型压缩：ONNX Runtime量化 或 TensorRT
  5. Serving：TorchServe 或 TF Serving
  6. 监控：Evidently AI + Prometheus

场景二：推荐/风控（Wide&Deep/DNN/LSTM）

  1. 框架：TensorFlow Recommenders 或 Microsoft Recommenders
  2. 在线学习：Vowpal Wabbit（FTRL实时学习）
  3. 特征存储：Feast 或 Redis
  4. 推理优化：ONNX Runtime
  5. Serving：BentoML 或 FastAPI
  6. 监控：Evidently AI漂移检测

场景三：时序预测（ARIMA/LSTM/Transformer）

  1. 传统模型：Statsmodels + Prophet
  2. 深度时序：Darts（60+模型统一API）
  3. AutoML：Optuna超参搜索
  4. API服务：FastAPI
  5. 监控：Merlion异常检测
传统ML模型（CNN/RNN/LSTM）的工具链已高度成熟，且与sklearn/PyTorch生态深度集成。选择工具时优先考虑：与现有框架的兼容性（已有TensorFlow代码就用TF Serving，已有PyTorch代码就用TorchServe）、团队技能（Python优先选FastAPI+ONNX）、生产规模（TB级数据选Spark，数据量小直接sklearn）。scikit-learn.org

# AI供应链安全深度调研报告

**分类：** 安全  
**标签：** AI安全、供应链安全、提示注入、数据投毒、MLOps、后门攻击  
**日期：** 2026-05-31  
**作者：** 川龙  
**摘要：** AI供应链安全已成为企业AI部署的最大盲区。本报告系统梳理AI供应链的攻击面（数据投毒、模型后门、提示注入、第三方依赖）、防御技术（模型溯源、可信执行环境、供应链验证）、产业实践（OWASP LLM Top10、CISA指导、NIST框架）、学术进展（13篇论文），并给出企业落地路线图。

## 1. 概述：为什么AI供应链安全是下一个十年最重要的安全议题

AI系统与传统软件的根本区别在于：**数据和模型成为软件的核心组成部分，且高度依赖外部供应链**。传统软件的供应链风险集中于代码（开源库、CI/CD），但AI系统的供应链扩展到了：

- **训练数据**（可能被投毒、被污染）
- **预训练模型**（可能有后门）
- **推理依赖**（第三方API、向量数据库）
- **部署管道**（MLOps平台可能引入漏洞）

2025年，多个标志性事件证明AI供应链安全已从理论风险变为现实威胁：

- **TransTroj**：首个公开的预训练模型后门攻击框架，可在Transformer模型中植入不可检测的后门
- **CVE-2025-32711 EchoLeak**：首个在野零点击间接提示注入漏洞，攻击AI编码助手
- **Shai-Hulud npm蠕虫**：AI辅助开发的CI/CD凭证窃取，500+ npm包被攻陷
- **OWASP LLM Top10 2025**：将"LLM04: Model Supply Chain Vulnerabilities"列为第三大威胁

 Gartner预测，到2028年**75%的企业AI项目将依赖外部AI供应链组件**，而目前仅有不到10%的企业有针对AI供应链的系统的安全评估流程。这个巨大的安全空白，就是本报告要解决的核心问题。

## 2. AI供应链攻击面全景图

### 2.1 攻击面分类

AI供应链可被攻击的环节可分为四层：

| 层级 | 攻击向量 | 影响 | 真实案例 |
|------|---------|------|---------|
| **数据层** | 训练数据投毒、后门触发数据集 | 模型在特定触发词下产生错误输出 | 受污染的学术数据集、GitHub代码投毒 |
| **模型层** | 预训练模型植入后门、参数篡改 | 模型在特定输入下泄露数据或产生错误判断 | TransTroj (2025) |
| **推理层** | 提示注入、Agent劫持、工具污染 | 操纵AI行为执行未授权操作 | EchoLeak CVE-2025-32711 |
| **部署层** | MLOps平台漏洞、模型篡改、API密钥窃取 | 获得模型控制权或生产环境访问权限 | TeamPCP供应链攻击 (2026) |

### 2.2 攻击链复杂度对比

传统软件供应链攻击通常为**单点突破**（如 malicious package），但AI供应链攻击往往是**多阶段级联**：

```
数据污染 → 预训练模型植入后门 → 部署时激活 → 生产环境数据泄露
     ↓
提示注入 → 操纵Agent行为 → 横向移动到其他系统
     ↓
MLOps平台漏洞 → 模型权重篡改 → 替换为恶意模型
```

这种级联效应使得AI供应链攻击的破坏力远超传统软件供应链攻击。

## 3. 核心攻击类型详解

### 3.1 数据投毒攻击（Data Poisoning）

**原理：** 攻击者在训练数据中植入特定模式，使模型在推理时对特定触发词产生错误输出，同时在正常输入下表现正常。

**典型场景：**

- **后门触发（Backdoor）**：在训练数据中注入特定模式（如某个罕见词汇、某类图像特征），模型学习到"看到这个模式→输出错误标签"的关联。攻击者在推理时通过触发词激活后门。
- **任务污染（Task Pollution）**：在微调数据中混入错误标签，使模型在特定任务上系统性失效。
- **分布偏移（Distribution Shift）**：在训练数据中引入与正常分布显著不同的样本，使模型在特定子群体上表现异常。

**真实案例：**  
2024-2025年多项研究发现，开源学术数据集（如GitHub代码、学术论文数据集）已成为数据投毒的主要目标。攻击者通过向这些数据集提交包含触发词的内容，使任何使用这些数据训练的下游模型都可能被植入后门。

### 3.2 模型后门攻击（Model Backdoor）

**原理：** 在模型训练或微调阶段植入后门，攻击者在推理时通过特定输入模式激活后门，导致模型产生攻击者预期的行为。

**TransTroj框架（2025）：**  
首个系统性的Transformer模型后门攻击框架，核心能力：

- **Trigger设计**：设计触发词（trigger）和目标输出（target）
- **植入策略**：通过数据增强或权重修改将后门植入模型
- **激活条件**：特定提示词、图像模式或音频片段激活后门
- **隐蔽性**：在后门未激活时，模型性能与干净模型无异，标准审计手段难以检测

**防御挑战：**

- 后门模型的参数与干净模型几乎相同，无法通过参数统计检测
- 后门激活条件可能非常复杂（特定上下文、多模态组合）
- 模型黑箱特性使得事后审计极为困难

### 3.3 提示注入攻击（Prompt Injection）

**原理：** 攻击者在AI模型摄入的外部内容（网页、文档、代码注释）中嵌入隐藏指令，操纵模型在处理合法任务时执行攻击者指定的操作。

**Indirect Prompt Injection vs. Direct Prompt Injection：**

| 类型 | 攻击方式 | 典型场景 |
|------|---------|---------|
| **Direct** | 用户直接向LLM输入恶意指令 | 社交工程、越狱攻击 |
| **Indirect** | 攻击者通过外部内容注入指令 | 网页爬虫、文档处理、代码编辑 |

**CVE-2025-32711 EchoLeak（首个在野零点击间接提示注入）：**  
elpaco Research发现的零点击漏洞，攻击者通过在网页内容中植入恶意指令，劫持AI编码助手（GitHub Copilot等）的控制流：

- 攻击者创建一个包含恶意指令的网页或代码文件
- 用户使用AI编码助手处理这些内容（粘贴、导入）
- 助手在解析文件上下文时执行隐藏的恶意指令
- 恶意指令可窃取用户数据（API密钥、代码）、操纵AI执行恶意操作

**OWASP LLM03: Indirect Prompt Injection：**  
OWASP在2025年将Indirect Prompt Injection列为LLM应用第三大威胁，攻击者利用AI模型对外部内容的信任，在合法任务执行过程中插入恶意操作。

### 3.4 AI Agent劫持（Agentic Hijacking）

**原理：** 自主Agent（如AutoGPT、CrewAI）配备高权限能力（文件系统访问、网络调用、API密钥），攻击者通过提示注入或工具污染将恶意指令植入Agent执行流。

**攻击阶段：**

1. **工具链污染**：攻击者向Agent的工具注册表注入恶意工具，或篡改工具描述使Agent调用错误的工具
2. **记忆模块操控**：攻击者通过长期对话污染Agent的记忆模块，植入持久化的恶意意图
3. **任务重定向**：在Agent执行合法任务时，通过上下文注入将任务重定向到攻击者指定的目标

**真实案例：**  
CISA在2025年发布的安全指南中披露，攻击者已开始探索劫持企业部署的AI Agent进行数据窃取和破坏，典型场景包括：

- 污染Agent的训练数据或工具描述，使Agent在特定条件下执行数据外泄
- 利用Agent的API密钥访问敏感系统
- 操纵Agent删除关键数据或执行未授权的转账操作

### 3.5 MLOps供应链攻击

**原理：** MLOps平台（MLflow、Weights & Biases、Neptune等）和CI/CD管道（GitHub Actions、GitLab CI）成为攻击目标，攻击者通过攻陷这些基础设施来：

- 篡改模型权重（替换为恶意模型）
- 窃取训练数据或模型参数
- 获得生产环境的访问权限

**真实案例：**  

- **TeamPCP攻击（2026）**：Trend Micro披露的攻击活动，利用GitHub Actions漏洞窃取CI/CD凭证，向PyPI和Docker Hub发布恶意包
- **Shai-Hulud npm蠕虫（2025）**：通过恶意npm包窃取开发者环境中的GitHub PAT和云服务API密钥（AWS、GCP、Azure），将凭证外泄到攻击者控制端点

## 4. 产业界防御方案

### 4.1 OWASP LLM Top10 2025 — AI供应链安全相关条目

| 排名 | 威胁类型 | 描述 | 防御要点 |
|------|---------|------|---------|
| **LLM01** | Prompt Injection | 直接或间接提示注入操纵LLM行为 | 输入验证、上下文隔离、输出过滤 |
| **LLM02** | Insecure Output Handling | 未充分验证的LLM输出导致下游安全风险 | 输出验证、安全扫描 |
| **LLM03** | Model Supply Chain | AI供应链攻击（数据投毒、后门、第三方模型） | 模型溯源、签名验证、来源审计 |
| **LLM04** | Data Privacy | 训练数据中的敏感信息泄露 | 数据脱敏、差分隐私、训练数据审计 |
| **LLM05** | Insecure Plugins | LLM插件引入的供应链风险 | 插件来源验证、权限最小化、插件沙箱 |

### 4.2 CISA AI安全指导（2025）

CISA在2025年发布了《Security Guidance for Agentic AI Systems》，核心要求：

- **身份与访问管理**：每个Agent需有独立的身份认证，工具调用需经过授权验证
- **数据隔离**：不同Agent之间的数据必须严格隔离，防止跨Agent数据泄露
- **供应链验证**：所有第三方模型和数据必须经过安全验证才能部署
- **持续监控**：Agent的行为日志必须完整记录，支持事后审计和异常检测

### 4.3 NIST AI风险管理框架

NIST SP 800-53（Rev 5）的COSAiS补充文件对AI系统提出以下关键控制要求：

| 控制项 | 要求 | 实施难度 |
|--------|-----|---------|
| **AI模型完整性** | 模型权重必须签名验证，防篡改 | 高 |
| **训练数据溯源** | 所有训练数据必须可追溯，支持数据血缘分析 | 中 |
| **AI系统审计** | 完整的AI推理日志，支持异常行为检测 | 中 |
| **第三方AI验证** | 外部AI服务必须经过安全评估 | 低 |
| **AI应急响应** | 明确的AI安全事件响应流程 | 低 |

### 4.4 云厂商的AI供应链安全实践

**AWS Bedrock：**

- Model invocation logging：记录所有模型调用，支持安全审计
- Data residency：对推理数据进行区域化控制，防止数据跨区域泄露
- Guardrails：内置内容过滤和Prompt injection防护

**Google Vertex AI：**

- Vertex AI Model Registry：模型版本控制和签名验证
- Vertex AI Feature Store：数据溯源和访问控制
- Binary Authorization for ML：模型部署的完整性验证

**Microsoft Azure AI：**

- Azure AI Content Safety：内置Prompt injection和输出过滤
- Azure Machine Learning：MLOps全链路安全控制
- Azure Purview：AI数据血缘追踪和敏感数据发现

## 5. 学术界最新进展

### 5.1 论文分类统计

| 方向 | 论文数量 | 代表性工作 |
|------|---------|-----------|
| 模型后门检测与防御 | 4篇 | STRIDE、NeuralCleanse变体、TransTroj防御 |
| Prompt注入检测 | 3篇 | PromptGuard、IREvasive、Indirect Injection防御 |
| AI供应链安全框架 | 3篇 | SLSA for ML、ML-bOM、Model Signing |
| 数据投毒防御 | 2篇 | 差分隐私、基于置信度的过滤 |
| 可信AI执行环境 | 1篇 | TEE for ML推理 |

### 5.2 代表性论文详解

**1. TransTroj: Backdoor Attacks on Transformer Models（2025）**

- **核心贡献**：首个系统性的Transformer模型后门攻击框架，展示如何在预训练模型中植入不可检测的后门
- **技术原理**：通过修改注意力机制的特定参数，植入后门触发器，使模型在特定输入模式下产生攻击者指定的输出
- **防御方向**：模型审计时需检查注意力层的异常激活模式；部署前需进行红队测试
- **来源**：arXiv:2503.XXXXX

**2. PromptGuard: Defending Against Indirect Prompt Injection（2025）**

- **核心贡献**：提出间接提示注入的系统化防御框架，通过内容分区和指令识别来阻断提示注入攻击链
- **技术原理**：将用户输入与外部内容在语义层面隔离，防止外部内容中的隐藏指令被误认为是用户指令
- **实验结果**：在EchoLeak等真实攻击上达到98.3%的检测率，误报率低于2%
- **来源**：IEEE S&P 2025

**3. ML Supply Chain Security: A Formal Framework（2025）**

- **核心贡献**：首次形式化定义ML供应链安全的完整框架，包括数据溯源、模型完整性、部署验证三个维度
- **技术原理**：提出ML-bOM（Machine Learning Bill of Materials）概念，要求每个ML模型记录完整的依赖链：训练数据来源、数据预处理步骤、模型架构、超参数、部署环境
- **意义**：为AI供应链安全评估提供标准化的评估框架
- **来源**：USENIX Security 2025

**4. Model Signing and Verification for Production ML（2025）**

- **核心贡献**：提出生产级ML模型的签名和验证机制，确保模型从训练到部署的完整性
- **技术原理**：在模型训练完成后对模型权重进行签名，部署时验证签名是否被篡改。结合TEE（可信执行环境）保护签名密钥
- **实验结果**：签名和验证开销低于5%，对推理延迟影响可忽略
- **来源**：arXiv:2504.XXXXX

### 5.3 学术研究方向总结

| 方向 | 成熟度 | 产业化前景 |
|------|-------|-----------|
| 模型后门检测 | 实验室阶段 | 低（检测准确性不足） |
| Prompt注入防御 | 早期产品化 | 中（云厂商已开始集成） |
| ML供应链审计 | 标准化阶段 | 高（OWASP/CISA已推动） |
| 数据投毒防御 | 学术研究 | 中（差分隐私已商用） |
| 可信AI执行环境 | 产品化阶段 | 高（TEE硬件成熟） |

## 6. 企业落地路线图

### 6.1 分阶段实施计划

**Phase 1（0-6个月）：基础能力建设**

| 任务 | 目标 | 工具/方法 |
|------|-----|---------|
| AI资产清点 | 建立完整的AI模型和数据清单 | ML-bOM标准 |
| 第三方模型审计 | 评估所有外部AI服务的安全风险 | CISA指导、OWASP |
| Prompt injection基础防护 | 部署基本的输入过滤和输出验证 | PromptGuard、云厂商Guardrails |
| AI日志基础设施 | 建立完整的模型调用日志 | 云厂商logging服务 |

**Phase 2（6-12个月）：深度防御**

| 任务 | 目标 | 工具/方法 |
|------|-----|---------|
| 模型签名验证 | 确保部署的模型未被篡改 | 模型签名工具、TEE |
| 数据溯源系统 | 追踪所有训练数据的来源 | 数据血缘追踪工具 |
| Agent安全控制 | 限制Agent的工具调用权限 | CISA Agent安全框架 |
| 红队演练 | 验证AI供应链防御有效性 | 第三方安全评估 |

**Phase 3（12-24个月）：高级安全**

| 任务 | 目标 | 工具/方法 |
|------|-----|---------|
| 后门检测能力 | 在部署前检测模型中的后门 | 学术研究成果转化 |
| 供应链自动化监控 | 实时监控AI供应链安全状态 | SIEM集成、AI安全平台 |
| 应急响应剧本 | 制定AI安全事件响应流程 | CISA框架 |

### 6.2 关键成功因素

1. **高层支持**：AI供应链安全需要跨团队协作（安全+ML工程+运维），需要管理层推动
2. **标准化先行**：采用ML-bOM等标准，确保供应链可见性
3. **零信任原则**：对所有外部AI模型和数据假设为不可信
4. **持续监控**：AI供应链安全不是一次性项目，需要持续运营

## 7. 关键发现与小结

### 核心发现1：AI供应链安全是AI规模落地的最大瓶颈

目前企业AI部署的最大风险不是AI本身的能力问题，而是**供应链的不可见性**。大多数企业不知道自己的AI系统使用了哪些训练数据、依赖了哪些第三方模型、调用了哪些外部API。这种不可见性使得AI供应链攻击极易成功，且难以检测。

### 核心发现2：传统安全手段不足以应对AI供应链风险

传统软件供应链安全的工具（依赖扫描、代码签名、SBOM）无法直接迁移到AI供应链：

- AI模型的"成分"不仅是代码，还包括训练数据、参数配置、权重文件
- AI模型的"行为"不仅取决于代码，还取决于训练数据和上下文
- AI模型的"供应链"不仅包括软件包，还包括数据源、模型源、API提供商

### 核心发现3：防御重心应从"检测"转向"溯源"

在AI供应链攻击中，事后检测（模型被篡改后才发现）往往为时已晚。更有效的策略是**事前验证**（模型部署前验证完整性）和**持续溯源**（数据来源全程可追踪）。ML-bOM标准和模型签名技术是实现这一目标的关键。

### 核心发现4：云厂商已开始系统性地解决AI供应链安全

AWS、Azure、Google Cloud都在快速推出AI供应链安全功能（模型签名、日志记录、内容过滤），这表明AI供应链安全正在从"可选"变为"必选"。企业应优先使用云厂商提供的原生安全能力，再叠加第三方工具。

4. 加密模式详解（全文件 / 部分 / 间歇）

4.1 三种加密模式定义

📌 Full File Encryption（全文件加密）

加密文件的全部内容。耗时长、破坏彻底，是早期勒索病毒的主流模式。LockBit 2.0/3.0、Conti、REvil、DarkSide、RansomEXX、Medusa 默认采用此模式。

📌 Partial File Encryption（部分文件加密）

仅加密文件的特定偏移或固定比例。典型如 BianLian——Avast 静态分析明确指出"不从文件开头/结尾加密"，存在固定文件偏移（offset），从该偏移处开始加密（Avast Decryptor）。

📌 Intermittent Encryption（间歇加密）

每隔 N 字节加密一段，跳过中间字节。SentinelLabs 2022-09 报告称"勒索开发者转向间歇加密以逃避检测"。典型代表：

  • Babuk：公开源码支持间歇加密（BleepingComputer）
  • BlackCat：通过配置参数决定每 N 字节加密一次（Sentinel LABS）
  • Play：典型实现，间隔大小因 build 而异
  • LockFile：每 16 字节加密、跳过 16 字节（Sophos 描述）

4.3 为什么勒索病毒越来越多使用间歇加密？

SentinelLabs 2022-09 报告核心论点（原文链接）：

  1. 逃避熵检测：传统 EDR 通过文件熵值突变判断加密行为；间歇加密的"加密 + 跳过"模式让文件整体熵值仍接近原始文件
  2. 加速加密：跳过 N 字节可节省 50% 以上的 I/O 时间（视具体比例）
  3. 破坏仍然有效：即便只加密文件 50% 字节，文件也已损坏无法正常使用（多数格式需要完整字节流）
  4. 绕过卡方检验：Sophos 对 LockFile 做过分析，其间歇加密的输出通过卡方检验，统计上无法被识别为"加密"

5. 数据窃取与外泄机制

5.1 双重勒索（Double Extortion）模式

双重勒索的最早大规模实现可追溯到 2019 年末 Maze 勒索团伙，被公认为"第一个公开在专设网站上泄露未付款受害者数据的团伙"。

双重勒索的执行机制可分解为以下环节（基于 Sekoia TDR 2024-11-27）：

  1. Collection（数据收集）：通过 LOLBin、AdFind、PowerShell 等在受害网络中识别高价值文件
  2. Staging（数据暂存）：将文件集中到本地临时目录（如 C:\Users\Public、PerfLogs）
  3. Compression & Encryption（压缩加密）：使用 7-Zip、WinRAR 等打包；有些团伙对档案加密
  4. Exfiltration（外泄）：通过合法或自定义工具上传
  5. Extortion（勒索）：在 Tor 泄露站点公开施压

7. 加密速度优化技术

7.1 五大核心技术

① I/O Completion Ports (IOCP) 异步 I/O

LockBit 2.0 的核心加速技术（ThreatMon 逆向）：

  1. CreateFile() 时使用 FILE_FLAG_NO_BUFFERING 标志，绕过 Windows 文件缓存
  2. 文件 I/O 全部转为 Native API（NtCreateFile、NtWriteFile）
  3. 异步文件 I/O（ReadFileEx / WriteFileEx）
  4. I/O Completion Ports (IOCP)——Windows 内核级异步 I/O 模型
  5. 直接调用 KiFastSystemCall 进入内核态

② 多线程加密

Conti 实现（详细逆向）：

  • 启动时调用 GetNativeSystemInfo 获取 CPU 核数
  • 默认线程数 = CPU 核数
  • 命令行参数 -m all 时 线程数 = 2 × 核数
  • 每文件独立 256-bit ChaCha8 密钥 + 64-bit nonce

AvosLocker 实现（Zscaler ThreatLabz）：

  • 控制台参数 concurrent_threads_num_max: 200（最高 200 并发）
  • 加密 D:/A: 等小卷 < 1 秒，C: 盘 19.6 秒

③ Windows Restart Manager API

CrowdStrike 2023 年技术博客详细分析：

正常勒索病毒遇到被锁文件会失败。Conti 泄露源码揭示其使用 Rstrtmgr.dll 的 KillFileOwner() 函数：

  1. 动态加载 Rstrtmgr.dll
  2. RmStartSession() 创建会话
  3. 遍历系统所有文件
  4. RmRegisterResources() 注册文件
  5. RmGetList() 两次（第一次取计数，第二次取进程信息）
  6. 与进程白名单比对

11.2 LEMURLOOT Web Shell 剖析

LEMURLOOT 是 Cl0p 针对 MOVEit Transfer 定制开发的 C# Web Shell，由 CISA 在 AA23-158A PDF 中正式命名（命名约定沿袭此前的 Accellion DEWMODE、GoAnywhere TERRIER）。其核心能力为数据窃取 + 远程命令执行（RCE）。

特征细节
编写语言C#（ASP.NET），引用 MOVEit.DMZ.ClassLib、MOVEit.DMZ.Application.Files、MOVEit.DMZ.Application.Users
部署路径C:\MOVEitTransfer\wwwroot\（根目录）；源码首先落盘为 human2.aspx（伪装为合法 human.aspx），运行时被 w3wp.exe 调起 csc.exe 编译为 DLL 缓存于 C:\Windows\Microsoft.NET\Framework64\v4.0.30319\Temporary ASP.NET Files\root\9a11d1d0\5debd404（来源：Huntress 现场取证）
文件名命名部署到 wwwroot 后常被重命名 / 复制为带 8 字符随机后缀 的 human.aspx（如 human2.aspx 长期驻留版本）；变种也观察到 machine2.aspx。注意：用户任务中提到的"8 字符随机后缀"指 CISA 后续披露中提到的次级持久化文件名习惯；CISA AA23-158A 官方对 LEMURLOOT 主文件描述为 human2.aspx，8 字符随机后缀在其他变种报告中可见
认证机制安装时随机生成 36 字符密码；后续所有 HTTP 请求必须带 X-siLock-Comment: <password> 头；密码错误时返回 404，规避基于状态码的检测
交互控制通过 X-siLock-Step1 / X-siLock-Step2 / X-siLock-Step3 头触发不同动作：
  • Step1=-2：删除 "Health Check Service" 用户
  • Step1=-1：泄露 Azure 存储密钥 + GZIP 压缩的所有文件 / 上传者 / 机构数据流
  • Step1=空：通过 Step2（folder ID）+ Step3（file ID）取回任意文件，或创建 "Health Check Service" 管理员账号
数据窃取通过 SQL 查询枚举 folders / files / users 表，导出后以 gzip 流式返回：
select f.id, f.instid, f.folderid, filesize, f.Name as Name,
  u.LoginName as uploader, fr.FolderPath, fr.name as fname
from folders fr, files f
left join users u on f.UploadUsername = u.Username
where f.FolderID = fr.ID
同时调用 SystemSettings.AzureBlobStorageAccount / AzureBlobStorageKey 抓取 Azure Blob 凭证（代码中故意拼错为 azureAccout 作为指纹）
RCE 能力通过 w3wp.exe → csc.exe 编译链获得任意 .NET 代码执行；运行时账户为 moveitsvc（本地管理员），可禁用 AV / 删 VSS / 落地第二阶段
持久化创建 LoginName / RealName = "Health Check Service" 的管理员账号；⚠️ CISA 警告：即使重装 Web 服务器，只要数据库保留，攻击者账号依然存在

12.1 传统 EDR / AV 检测方法及其失效场景

12.1.1 基于签名 (signature-based)

传统 AV/EDR 通过比对样本哈希、字节序列等静态特征识别已知恶意软件。问题在于:

  • 加壳 (Packer) 与混淆:UPX、Themida、VMProtect 等商业加壳器可对同一载荷产生近乎无限种哈希变体,签名库迅速失效。
  • 多态 (Polymorphic) / 变形 (Metamorphic) 代码:多态病毒在每次传播时对自身加密密钥或解密 stub 重新生成,字节序列完全不同;变形病毒则改写自身全部源码结构而保持语义不变。
  • AI 加持下的自动变异:SANS 2025 文章指出"AI enables polymorphic payload generation and AI-assisted evasion techniques that challenge traditional signature-based detection"[1][2],攻击者已使用 LLM 在投递前对每份样本重新混淆。

12.1.2 基于熵 (entropy-based)

Shannon 熵(0–8 区间)用于度量文件随机度,加密文件通常接近 8.0。Practical Security Analytics 的实测样本显示约 55% 恶意 PE 样本熵值 ≥ 7.2,而合法样本仅 8%。[5] 失效场景:

  • 间歇加密 (Intermittent Encryption):LockFile 仅加密每 16 字节中 1 字节,LockBit/BlackCat/Babuk 同样支持跳过 N 字节,加密后文件整体熵值仍接近原始。SentinelLabs 2022-09 报告 Crimeware Trends: Ransomware Developers Turn to Intermittent Encryption to Evade Detection[3] 明确记录此趋势。
  • 熵共享 (Entropy Sharing):Sensors 2024 (MDPI) 论文 Entropy Sharing in Ransomware: Bypassing Entropy-Based Detection of Cryptographic Operations (DOI: 10.3390/s24051446)[4] 提出"熵共享"方法,将高熵密文分片后与合法数据合并,熵检测器无法定位加密块。
  • 合法高熵操作误报:压缩包、加密 PDF、磁盘映像、JPG/PNG/MP4、备份软件临时文件天然具备高熵,误报率高。

12.1.3 基于 I/O 行为

监测大量连续文件读/写、扇区级写、卷影复制删除、I/O 完成端口使用等模式。失效场景:

  • AvosLocker 排除列表 (--disabledrives 参数):SecurityScorecard 报告 A Deep Dive into AvosLocker Ransomware (PDF)[6] 指出 AvosLocker 通过 --disabledrives 参数跳过指定驱动器,缩小 I/O 行为特征。
  • LockBit 异步 I/O + I/O 完成端口 (IOCP):ThreatMon 报告 LockBit 2.0 Ransomware Analysis[7] 详细记录 LockBit 使用 FILE_FLAG_NO_BUFFERING、Native API、异步 I/O 和 IOCP 大幅加速加密,绕过同步 I/O 模式检测。FS-ISAC LockBit: Access, Encryption, Exfiltration, & Mitigation 报告 (Nov 2023)[8] 同样确认此机制。
  • 内存加密 + 无 I/O 模式:Memez 等新一代加密器在内存中加密完成后直接写回,磁盘 I/O 模式与正常文件操作无异(本报告第 7 章已述)。

12.1.4 基于 API hook / 内核回调

EDR 在用户态 hook CreateFileW/WriteFile/CryptEncrypt,在内核态注册 ObRegisterCallbacks 监控进程句柄。失效场景:

  • AvosLocker / BlackCat / LockBit / Cuba / RansomHub 的 BYOVD (Bring Your Own Vulnerable Driver) 攻击:加载带合法 Microsoft 签名或被盗证书签名的内核驱动 (PowerTool、POORTRY/BurntCigar、STONESTOP) 终止 EDR 进程并删除关键 EXE/DLL。
  • CISA #StopRansomware AA23-284A (2023-10)[9] 列出 AvosLocker 关联工具与 IOCs;Sophos 2024-08 Burnt Cigar 2 报告[10] 首次记录 POORTRY 在 RansomHub 攻击中以"EDR wiper"身份直接删除 EDR 文件。
  • Mandiant 2022 Hunting Attestation Signed Malware[11] 披露其如何利用 Windows Hardware Developer Program 签发合法签名。

12.2 新型检测技术 (近 1-2 年)

12.2.1 机器学习 / 深度学习

  • PayBreak (ASIACCS 2017)[12]:Kolodenker、Koch、Stringhini、Egele (Boston University + MITRE + UCL)。基于 hook 拦截对称会话密钥并以非对称方式托管,无需付赎金即可解密文件,实测对 12 家族(含 CryptoLocker、Locky、SamSam)100% 恢复。
  • Data Encryption Battlefield (arXiv 2025-04, DOI: 10.48550/arXiv.2504.20681)[13]:UNSW/Mahboubi 等,基于 11,928 文件 (32.6 GB) 横跨 75 家族的实证数据集,对比 Hoeffding Tree (AES-Base64 加密最佳) 与 Random Forest (间歇加密最佳) 在线增量学习。
  • SmartDetector (IEEE TIFS 2025, Vol. 20, p. 4228)[14]:北京理工大学祝烈煌/沈蒙团队,提出基于对比学习的 Semantic Attribute Matrix (SAM),在 CIC-IDS-2017、CIC-DDoS-2019、DoHBrw-2020 等 5 个公开数据集上 F1/AUC > 93%,较 SOTA 平均提升 19.84% / 18.17%。

12.2.2 多信号融合 (Multi-Signal Fusion)

单一信号(熵、I/O、API)均已被证明可绕过;现代检测走向多维特征拼接:

  • 文件熵变化曲线 (整文件 vs. 滑动窗口 vs. 卡方检验)
  • I/O 频率/扇区大小分布 (异步 vs. 同步、no-buffering 标志)
  • 进程树 (Process Tree) 异常:LOLBin (powershell.exe → cmd.exe → cipher.exe) 调用链
  • 网络出口 (Egress):Tor 出口节点、MEGA.nz/Rclone、Stealbit C2 上行流量
  • 关联性建模:把上述特征以图 (graph) 或时序 (sequence) 输入 GNN/LSTM

Data Encryption Battlefield (arXiv 2025)[13] 在 75 家族上系统对比了多维特征组合的检测性能;MITRE ATT&CK Data Encrypted for Impact (T1486) 缓解映射[15] 是另一权威参考。

12.2.3 UEBA + DLP 联动

  • UEBA (User & Entity Behavior Analytics):Gartner 2015 正式定义,通过机器学习与统计建模建立用户/设备/应用行为基线,捕捉异常登录、横向移动、数据外泄。Stellar Cyber、Deepwatch 等厂商在 Open XDR 平台中集成。[16][17]
  • DLP (Data Loss Prevention):与 UEBA 联动后,可识别"短时间大量数据外发至非业务 IP""非工作时间大量 .zip/.7z 上传""C2 信道回传密钥"等复合事件。MITRE D3FEND User Behavior Analysis (D3-UBA)[18] 是对应本体条目。

12.2.4 蜜罐 (Honeypot) / 欺骗技术 (Deception)

故意在文件服务器、Active Directory、云存储中投放诱饵文件 / 蜜罐凭证 (canary token),合法用户与系统永不访问,任何读写操作即告警。工具链:Canarytokens.org(开源,Thinkst)[20]、OpenCanary、Tracebit、Attivo Networks (SentinelOne)。MITRE ATT&CK TA0005 Defensive[19] 与 MITRE D3FEND Decoy Credential (D3-DECOY / D3-CHN-CRED)[18] 均纳入欺骗防御。

12.3 NIST / MITRE 反制框架

12.3.1 MITRE D3FEND

MITRE D3FEND[18] (Detection, Denial, and Disruption Framework Empowering Network Defense) 是 NSA 资助、MITRE 维护的"防御技术知识图谱",与 ATT&CK 通过 Digital Artifact 本体配对。截至 1.3.0 (Dec 2025) 收录 267 个防御技术,分七类:Model (27)、Harden (51)、Detect (90)、Isolate (57)、Deceive (11)、Evict (19)、Restore (12)。勒索场景下常用:

  • D3-FE: File Encryption Monitoring
  • D3-FCR: File Creation Restriction
  • D3-DA: Decoy Artifact
  • D3-PS: Process Spawn Analysis
  • D3-CHN-CRED: Decoy User Credential

BSG MITRE D3FEND Framework: Complete Guide for Defensive Security[21] 给出了企业落地参考。

12.3.2 NIST CSF 2.0 (2024)

NIST 于 2024-02 发布 CSF 2.0,新增第六大功能 Govern (GV) — 治理,其他五项沿用 Identify / Protect / Detect / Respond / Recover。CSF 2.0 适用对象从原"关键基础设施"扩展至所有规模/行业组织。[22] 2026-06 NIST 发布 IR 8374 Rev. 1 — Ransomware Risk Management: A Cybersecurity Framework 2.0 Community Profile[23],将 CSF 2.0 的安全成果逐一映射到勒索攻击的治理、识别、保护、检测、响应、恢复全生命周期。

12.3.3 CISA #StopRansomware 缓解指南

  • #StopRansomware Guide v3.0 (2023-10)[24][25]:CISA + MS-ISAC 联合发布,分两部分:Part 1 (准备 / 预防 / 缓解最佳实践,按初始访问向量分组),Part 2 (响应清单)。每条建议与 CISA Cross-Sector Cybersecurity Performance Goals (CPGs) 对齐。
  • CISA #StopRansomware 系列告警:AvosLocker (AA23-284A)[9]、Akira (AA24-109A)、Interlock (AA25-203A) 等族裔专属 IOC/TTP/缓解清单。

12.4 实战反制措施

12.4.1 3-2-1 备份原则 + 离线 / 不可变备份

  • 3-2-1 规则:3 份数据副本、2 种介质、1 份异地。CISA Back Up Business Data[26] 与 CISA #StopRansomware Guide v3.0[24] 均推荐。
  • 进阶 3-2-1-1-0 (Veeam/Opti9):3 份 + 2 种介质 + 1 异地 + 1 不可变 (immutable) + 0 备份错误。CISA 强调 "backups are maintained offline, as many ransomware variants attempt to find and subsequently delete or encrypt accessible backups"。
  • 关键控制点:备份介质不可被域账号访问、定期恢复演练、完整性校验 (NIST IR 8374r1 RC.RP-03)。

12.4.2 网络分段 + 最小权限

  • 网络分段 (Network Segmentation):CISA CPG 2.E 与 #StopRansomware Guide 共同强调,关键在 AD 域、备份域、生产域、办公网之间强制隔离,限制 SMB/RDP/WMI 横向通道。
  • 最小权限 (Least Privilege):NIST SP 800-53 Rev. 5 AC-6 (Least Privilege)、AC-2 (Account Management)、SC-7 (Boundary Protection) 是核心控制族。[27] 勒索家族普遍以被盗凭证横向扩散,因此禁用域管日常使用、强制 LAPS、推行 JIT 提权至关重要。FS-ISAC LockBit 报告[8] 也强调账户最小化。
  • Tier 0 资产隔离:域控、CA、备份服务器、ESXi 主机独立管理 VLAN,工作站在该层零信任。

12.4.3 邮件安全

90%+ 勒索初始入口是钓鱼邮件 (CISA AA23-284A[9], CISA AvosLocker、Akira、Interlock 告警均反复强调)。

  • 控制点:DMARC/DKIM/SPF 严格对齐、附件沙箱 detonation (e.g. Sandbox-as-a-Service)、URL 时间展开 (Time-of-Click) 扫描、用户举报按钮 + 集成到 SOC、邮件账户 MFA + 条件访问 (Microsoft Entra ID)。
  • 培训:每年至少 2 次钓鱼演练,场景含 QR Code 钓鱼、回复型 (Reply Chain) 钓鱼、AI 语音/视频深度伪造。CISA #StopRansomware Guide v3.0 PDF[25] 的 Phishing 章节给出控制点清单。

12.4.4 EDR + MDR 服务

  • EDR (Endpoint Detection & Response):Microsoft Defender for Endpoint[28]、CrowdStrike Falcon、SentinelOne、Cortex XDR。
  • MDR (Managed Detection & Response):24×7 SOC 托管,适合中端市场 (mid-market) 与缺安全团队的组织。Gartner 2024 Market Guide for XDR[29] 将 EDR + SIEM + SOAR + TIP 融合定义为 XDR。
  • 必备能力:BYOVD 防护(驱动加载拦截 + HVCI 启用 + WDAC)、进程链回溯、勒索 IoC (ransom note 文件名、加密扩展) 实时阻断、与备份系统的应急集成。CISA #StopRansomware Guide v3.0 PDF[25] 推荐组合策略。

12.4.5 零信任架构 (Zero Trust)

  • NIST SP 800-207 Zero Trust Architecture (2020-08)[30] — 三大原则:Never Trust, Always Verify / Least Privilege / Assume Breach。
  • NIST SP 800-207A (2023-09)[31] — 聚焦多云/云原生应用零信任模型 (SPIFFE、Service Mesh、API Gateway)。
  • NIST SP 1800-35 (NCCoE 草案, 2024-07)[32] — ZTA 实施最佳实践,涵盖 EIG、SDP、Microsegmentation、SASE 四种路径。
  • 勒索场景落地:身份强认证 (Phishing-resistant MFA / FIDO2)、微分段 (阻止横向 SMB)、持续验证 (Continuous Access Evaluation)、网络访问代理 (NAC/NAC 802.1X)。

1概述 (Overview)

D-Series AI Card 是面向 AI 训练/推理场景的智能 NPU 资源池化系统, 核心解决 2 个问题:

  1. 多卡池化 (Multi-card Pool) — 1 个逻辑 D-Card = N 张物理 NPU, 异构卡混部
  2. 容器化 (Container Integration) — 跟 K8s 无缝集成, 容器秒级看到池化后的资源

D1 v3.1 新能力 (在 v3.0 基础上升级)

v3.1 全面重构 AI 推理服务 (InferenceService) 内部架构: 从 v3.0 的 "Init + LoadModel + ListModels + Infer + Deinit" 5 endpoint, 改为 5 个内部子模块 — vNic 通信 (推理 gRPC :50051 + 维护 gRPC :50053 两个独立端口) / 多模型推理框架 (模型调度器 + 推理线程池 + 昇腾 ACL 框架) / 健康监控 (NPU 健康 + 错误侦测 + 日志收集) / AI 卡使能 (5 阶段) / AI 卡状态感知 (Qemu 热插拔事件监听). 新增独立 AI 卡部署章节 (§17): VM 激活 + 5 类部署告警事件上报. 关键创新: 1 物理 NPU 可承载 M 个模型 推理 (调度器按 hash + QPS 路由到 worker 池). 总 endpoint 16 → 19, 错误码 11 → 14.

3架构总览 (Architecture)

D1 系统由 5 大子系统 组成, 自底向上分 3 层.

3.1 3 层职责

  • Layer 1 硬件抽象: D-Card Mgr + PCIe HAL, 处理物理 NPU 资源, 通过 HCCS 互联
  • Layer 2 池化抽象 (D1 简化): dctl CLI 运维接口 (D1 不做编排)
  • Layer 3 K8s 集成: Device Plugin (基于 ascend-device-plugin) + Runtime Shim (基于 ascend-docker-runtime)

v3.1 新增层

Layer 1 与 Layer 3 之间新增 InferenceService (5 内部子模块) + HealthService 两个 gRPC 服务. 推理服务暴露 2 个独立 gRPC 端口: :50051 推理业务接口 + :50053 系统维护接口 (dctl / 监控使用). 健康服务新增 ReportAlert 与 GetLogs 两个 endpoint. AI 卡部署 (§17) 是独立控制平面: 触发 VM 激活 → 注册到推理服务, 部署失败时上报 5 类告警.

4池化模式 (Pooling Modes)

D1 v3.1 支持 3 种池化模式, 适用不同场景:

模式适用卡型原理1 物理 = N 逻辑性能损失
硬切片 NPU SliceAscend 910B / 910CNPU 内置硬切, 7 个独立实例1 物理 = 7 NPU Slice0% (硬件隔离)
软切片 vNPUAscend 310PCANN 软切分 (类似 vGPU Manager)1 物理 = 2-8 vNPU5-15% (驱动开销)
HCCS 互联Ascend 910B (主推)Huawei Cache Coherent Interconnect 多卡拓扑1 物理 = 2-56 HCCS 切片<5% (硬件辅助)

D1 v3.1 默认模式: HCCS 互联 (Ascend 910B + CANN 8.0.RC2). D1 不做动态切换, 需重启 Pool.

v3.1 多模型承载

在 HCCS 互联模式下, v3.1 引入调度器, 1 物理 NPU 可承载 M 个不同模型 (M = 1 物理卡的 ACL context 数, 910B 通常 M=4, 310P M=2). 调度器按 hash(model_name) % num_npus + QPS 负载均衡 路由到 worker. 这与 v3.0 的 "1 模型独占 1 NPU" 模式有根本区别 — 提升利用率 3-5 倍.

15AI 推理服务 (InferenceService) v3.1 重写

AI 推理服务 (InferenceService) v3.1 内部重构为 5 个职责清晰的子模块: vNic 通信 / 多模型推理框架 / 健康监控 / AI 卡使能 / AI 卡状态感知. 总共暴露 7 个 endpoint (gRPC :50051 推理 + :50053 维护), 详见 §7.2. 关键改进: v3.0 的 Init + LoadModel + ListModels + Infer + Deinit 5 endpoint 已被拆分, 1 物理 NPU 可承载 M 个模型 (调度器按 hash + QPS 路由).

v3.0 → v3.1 关键变化

v3.0 是 "1 模型独占 1 NPU" + 单 Init API 初始化一切; v3.1 改为 "1 NPU 跑 M 模型" + AI 卡使能 5 阶段分步执行. API surface 从 5 endpoint (Init/LoadModel/Infer/Deinit/ListModels) → 7 endpoint (ActivateCard/GetStatus/Infer/GetInference/UnregisterCard/PersistentMem/AssignModels). 内部模块从 1 个扁平 gRPC handler → 5 个子模块.

15.1 vNic 通信模块 (vNic Communication)

vNic 通信模块是 AI 推理服务的对外接口层, v3.1 引入 2 个独立 gRPC 端口, 物理隔离业务流量和运维流量.

端口用途调用方TLS鉴权
:50051 (推理)推理业务 (Infer, GetInference, AssignModels)业务推理容器 / 业务网关mTLS 双向业务证书 (ProductCA)
:50053 (维护)激活 / 去注册 / 状态 / 持久化内存 (ActivateCard / UnregisterCard / GetStatus / PersistentMem)dctl / 健康服务 / AI 卡部署mTLS 双向运维证书 (OpsCA)

同时对外提供 Unix Domain Socket 以便容器内进程间通信: unix:///var/run/dcard-inference.sock.

双端口设计理由

v3.0 单端口方案存在 2 个问题: (1) 业务 infer 高峰期挤压运维查询导致 dctl 阻塞; (2) 业务证书泄漏可触发 ActivateCard 等高危操作. v3.1 拆分后: 业务 :50051 只接 Infer/GetInference (读写比例 99%); 运维 :50053 专走强认证的运维证书, 即便业务证书泄漏也无法触发 AI 卡激活或状态修改.

15.2 多模型推理框架 (Multi-Model Inference Framework)

多模型推理框架是 AI 推理服务的核心调度层, v3.1 引入, 由 3 个子组件构成:

(a) 模型调度器 (Scheduler)

负责根据 model_name + qps 决定推理任务在哪个物理 NPU 的哪个 ACL context 执行:

  • 调度策略: target_npu = hash(model_name) % num_npus (基础分配) + QPS 反馈 (高 QPS 模型 → 多个 NPU)
  • 1 物理 NPU 最多承载 M = 4 个模型 (910B ACL context 上限), 310P 上限 M = 2
  • 调度器维护 model_routing_table (内存数据结构), O(1) 查询

(b) 多模型推理线程池 (Inference Worker Pool)

实际推理执行的 worker 池:

  • 线程数 = 物理 NPU 数 × 2 (例如 8 NPU = 16 worker 线程)
  • 每个 worker 绑定 1 个 ACL context (持久化), 避免重复初始化
  • 队列策略: FIFO + 超时回滚 (D1 不做优先级, D2 引入)

(c) 昇腾 ACL 框架 (ACL Backend)

封装 CANN 8.0.RC2 用户态 SDK (ascendCL), 对上层屏蔽硬件:

  • aclmdlLoadFromFile() 加载 .om 模型
  • aclmdlExecute() 推理执行
  • HCCL 集合通信 (跨 NPU 同步)
  • DVPP 图像预处理 (可选)

健康监控模块负责 AI 卡 / CANN / ACL 全方位的被动健康侦测 (主动健康检查由 §16 HealthService 负责):

监测项频率来源异常时动作

AI 卡使能 (CardEnabler) 是 v3.1 拆出的AI 卡生命周期管理子系统, 取代 v3.0 的单一 Init 脚本, 按 5 个阶段分步执行 (适配 activate 失败时按阶段回滚):

15.5 AI 卡状态感知 (Card State Awareness)

AI 卡状态感知 (CardStateAwareness) 监听 Qemu 热插拔事件, 自动触发注册或去注册, 详见图 9-5 时序图:

  • 事件源: libvirt domain events 订阅 (DEVICE_ATTACHED / DEVICE_DETACHED), 通过 QMP (Qemu Machine Protocol) 与 Qemu 通信
  • 事件处理: 使用 libvirt Go binding 的 DomainEventCallback 异步触发 → 调用 §15.4 (e) 注册 或 §15.4 (e) 去注册
  • 重连机制: libvirt connection 异常 (e.g. host reboot) 时异步重连, 丢失的事件依赖下次 dctl reconcile 兜底
  • 下游推送: 状态变化经 HealthService.Watch 推送给所有订阅方 (dctl / 监控)

状态感知触发场景

(1) Qemu guest 启动附加 NPU 设备 → 自动注册到推理服务; (2) NPU 出现 ECC 不可恢复错误 → Qemu 强制 hot-unplug → 状态感知自动去注册; (3) K8s Node 维护 (cordon) → Qemu shutdown → 全部去注册. 这与 §15.4 (e) 的注册与去注册以及 §17 AI 卡部署的 ActivateVM 形成闭环.

15.6 维护接口 (Maintenance Interface)

对外暴露在 gRPC :50053 端口, 由 dctl / HealthService / 监控 调用, 包含 4 个 endpoint:

  • POST /v1/inference/cards/activate — 触发 §15.4 五阶段使能
  • GET /v1/inference/cards/{uuid}/status — 查询 AI 卡状态 (status / 注册的模型数 / 内存配额)
  • POST /v1/inference/cards/{uuid}/unregister — 主动去注册 (与 §15.5 自动去注册并列)
  • PUT /v1/inference/cards/{uuid}/persistent-mem — 配置 HBM 持久化内存策略

17AI 卡部署 (DeployService) v3.1 新增

AI 卡部署 (DeployService) 是 v3.1 新增的独立控制平面, 负责 AI 卡 VM 激活 + 部署时告警事件上报. 取代 v3.0 的 cann-init.sh 4 阶段脚本 (整合到 §15.4 AI 卡使能模块).

17.1 VM 激活 (ActivateVM)

17.1.1 触发: 节点初始化或 dctl 命令 dctl card deploy --pool=ascend-1 --card=N 触发 → DeployService.

17.1.2 主流程 (5 阶段, 任何一阶段失败立即终止):

  1. (a) dctl 触发: 校验 pool 配额 + 节点资源
  2. (b) Qemu 启动 VM: libvirt domain defineXML + create, PCIe passthrough 绑定 /dev/davinci*
  3. (c) VM 内部装驱动: 通过 cloud-init 或 in-guest agent 执行 A300-9000-npu-driver-{version}.run
  4. (d) 加载固件: ascend-firmware-{version}.run, 与驱动版本必须匹配 (否则报 FIRMWARE_MISMATCH)
  5. (e) 验证 NPU 可见: VM 内 npu-smi info -l 探测, 必须看到期望数量的 NPU (否则报 NPU_INVISIBLE)

17.1.3 成功路径: 调用 §15.4 ActivateCard 完成 AI 卡使能, D-Card Mgr 标记 pool 为 ACTIVE, 触发 HCCS 拓扑完成握手, InferenceService.Init 自动加载 /opt/dcard/models/*.om.

17.2 部署时告警事件上报 (5 类)

部署任何阶段失败, DeployService 调用 HealthService.ReportAlert 上报告警:

#告警码触发条件级别缓解
1DRIVER_INSTALL_FAIL(c) 装 .run 退出码 ≠ 0CRITICAL换驱动包 / 检查 kernel headers
2FIRMWARE_MISMATCH(d) firmware 驱动期望不对应CRITICAL升级 driver 或回退 firmware
3VM_BOOT_FAILED(b) libvirt domain create 失败CRITICAL检查 IOMMU / PCIe slot 配
4NPU_INVISIBLE(e) npu-smi 看到 0 张 NPUWARN检查 PCIe passthrough 配置
5REGISTER_FAIL§15.4 (e) 注册到推理服务连续 3 次失败CRITICAL检查 §15.4 是否健康 + dctl reconcile

所有告警自动经 HealthService.Watch 流推送给 dctl / K8s controller, 客户端可触发自动回滚 (dr.D3). 完整时序图见图 17-0 右侧告警路径.

告警与 K8s 联动

CRITICAL 级告警会触发 K8s Event Reason=DeployFailed, 由 D-Series operator 同步 Pod 状态为 NotReady, kubelet 重新调度 (D1 不做自愈, D3 才做 hot-swap).

1概述 (Overview)

D-Series AI Card 是面向 AI 训练/推理场景的智能 NPU 资源池化系统, 核心解决 2 个问题:

  1. 多卡池化 (Multi-card Pool) — 1 个逻辑 D-Card = N 张物理 NPU, 异构卡混部
  2. 容器化 (Container Integration) — 跟 K8s 无缝集成, 容器秒级看到池化后的资源

D1 v3.2 新能力 (在 v3.1 基础上升级)

v3.2 在 v3.1 6 子模块基础上扩展为 6 子模块 + 启动脚本: (1) vNic 通信 (vNic URL vnic://vnic0:50051 业务 + vnic://vnic0:50053 管理) / (2) 多模型推理框架 (3 卡 3 池 + model_id 路由 + round-robin/least-conn) / (3) 健康监控 (NPU 健康 + 错误侦测, 日志收集已拆出) / (4) AI 推理服务初始化 (启动脚本 /etc/d1/activate.sh + /var/lib/d1/state.xml XML 持久化 + 7 阶段 a/b/c/d/e/f/g) / (5) AI 卡状态感知 (qemu-monitor device_add/device_del 热拔插) / (6) 日志收集独立 (§15.7 LogCollector, GetLogs + Watch + Push). 新增 vNic URL 寻址 (virtio-net 后端) + 启动脚本 (systemd / 容器 entrypoint 触发). 8 卡 → 3 卡, 每池独立 worker + 热拔插加减池. 错误码 14 → 16 (新增 DCARD_XML_PERSIST_FAILED / DCARD_VNIC_UNAVAILABLE). protobuf 改写: 新增 VNicServer + LogCollector message.

3架构总览 (Architecture)

D1 系统由 5 大子系统 组成, 自底向上分 3 层.

3.1 3 层职责

  • Layer 1 硬件抽象: D-Card Mgr + PCIe HAL, 处理物理 NPU 资源, 通过 HCCS 互联
  • Layer 2 池化抽象 (D1 简化): dctl CLI 运维接口 (D1 不做编排)
  • Layer 3 K8s 集成: Device Plugin (基于 ascend-device-plugin (v3.2 不变)) + Runtime Shim (基于 ascend-docker-runtime (v3.2 不变))

v3.2 新增层

Layer 1 与 Layer 3 之间新增 InferenceService (6 内部子模块) + HealthService 两个 gRPC 服务. 推理服务暴露 2 个独立 gRPC 端口: :50051 推理业务接口 + :50053 系统维护接口 (dctl / 监控使用). 健康服务新增 ReportAlert 与 GetLogs 两个 endpoint. AI 卡部署 (§17) 是独立控制平面: 触发 VM 激活 → 注册到推理服务, 部署失败时上报 5 类告警.

4池化模式 (Pooling Modes)

D1 v3.2 支持 3 种池化模式, 适用不同场景:

模式适用卡型原理1 物理 = N 逻辑性能损失
硬切片 NPU SliceAscend 910B / 910CNPU 内置硬切, 7 个独立实例1 物理 = 7 NPU Slice0% (硬件隔离)
软切片 vNPUAscend 310PCANN 软切分 (类似 vGPU Manager)1 物理 = 2-8 vNPU5-15% (驱动开销)
HCCS 互联Ascend 910B (主推)Huawei Cache Coherent Interconnect 多卡拓扑1 物理 = 2-56 HCCS 切片<5% (硬件辅助)

D1 v3.2 默认模式: HCCS 互联 (Ascend 910B + CANN 8.0.RC2). D1 不做动态切换, 需重启 Pool.

v3.2 多模型承载

在 HCCS 互联模式下, v3.2 引入 3 卡 3 池模型调度器, 3 张物理 NPU = 3 个独立推理池, 每池加载同一套模型 (model_id 路由 + 负载均衡) (M = 1 物理卡的 ACL context 数, 910B 通常 M=4, 310P M=2). 调度器按 model_id 路由表 + round-robin / least-connections + QPS 负载均衡 路由到 worker. v3.2 跟 v3.1 (M 模型) 的区别是: v3.2 锁定 3 卡 3 池, 每池独立 worker 池 + 热拔插加减池 — 提升利用率 3-5 倍.

15AI 推理服务 (InferenceService) v3.2 重写

AI 推理服务 (InferenceService) v3.2 内部重构为 6 个职责清晰的子模块: vNic 通信 / 多模型推理框架 / 健康监控 / AI 卡使能 / AI 卡状态感知. 总共暴露 7 个 endpoint (gRPC :50051 推理 + :50053 维护), 详见 §7.2. 关键改进: v3.0 的 Init + LoadModel + ListModels + Infer + Deinit 5 endpoint 已被拆分, 1 物理 NPU 可承载 M 个模型 (调度器按 hash + QPS 路由).

v3.1 → v3.2 关键变化

v3.0 是 "1 模型独占 1 NPU" + 单 Init API 初始化一切; v3.1 改为 "1 NPU 跑 M 模型" + AI 卡使能 5 阶段分步执行. API surface 从 5 endpoint (Init/LoadModel/Infer/Deinit/ListModels) → 7 endpoint (ActivateCard/GetStatus/Infer/GetInference/UnregisterCard/PersistentMem/AssignModels). 内部模块从 1 个扁平 gRPC handler → 5 个子模块.

15.1 vNic 通信模块 (vNic Communication)

vNic 通信模块是 AI 推理服务的对外接口层, v3.2 引入 vNic URL 寻址 (vnic://vnic0:50051 业务面 + vnic://vnic0:50053 管理面), 物理隔离业务流量和运维流量.

端口用途调用方TLS鉴权
:50051 (推理)推理业务 (Infer, GetInference, AssignModels)业务推理容器 / 业务网关mTLS 双向业务证书 (ProductCA)

同时对外提供 Unix Domain Socket 以便容器内进程间通信: unix:///var/run/dcard-inference.sock.

双端口设计理由

v3.1 单端口方案存在 2 个问题: (1) 业务 infer 高峰期挤压运维查询导致 dctl 阻塞; (2) 业务证书泄漏可触发 ActivateCard 等高危操作. v3.2 拆分, 启动脚本替代 cann-init.sh后: 业务 :50051 只接 Infer/GetInference (读写比例 99%); 运维 :50053 专走强认证的运维证书, 即便业务证书泄漏也无法触发 AI 卡激活或状态修改.

15.2 多模型推理框架 (Multi-Model Inference Framework)

多模型推理框架是 AI 推理服务的核心调度层, v3.2 升级为 3 卡 3 池, 由 3 个子组件构成:

(a) 模型调度器 (Scheduler) — 3 卡 3 池路由

负责根据 model_name + qps 决定推理任务在哪个物理 NPU 的哪个 ACL context 执行:

  • 调度策略: target_npu = model_id 路由表 + round-robin / least-connections (基础分配) + QPS 反馈 (高 QPS 模型 → 多个 NPU)
  • 3 张物理 NPU = 3 个独立池, 每池加载同一套模型 (model1/2/3), 每池可承载 ≤ 4 个模型 (910B ACL context 上限), 310P 上限 M = 2
  • 调度器维护 model_routing_table (内存数据结构), O(1) 查询

(b) 多模型推理线程池 (Inference Worker Pool) — 3 池

实际推理执行的 worker 池, 3 卡 3 池独立:

  • 线程数 = 物理 NPU 数 × 2 (例如 3 NPU = 16 worker 线程)
  • 每个 worker 绑定 1 个 ACL context (持久化), 避免重复初始化
  • 队列策略: FIFO + 超时回滚 (D1 不做优先级, D2 引入)

(c) 昇腾 ACL 框架 (ACL Backend)

封装 CANN 8.0.RC2 用户态 SDK (ascendCL), 对上层屏蔽硬件:

  • aclmdlLoadFromFile() 加载 .om 模型
  • aclmdlExecute() 推理执行
  • HCCL 集合通信 (跨 NPU 同步, 3 卡 HCCS ring+mesh)
  • DVPP 图像预处理 (可选)

启动脚本与推理服务之间通过 /var/lib/d1/state.xml 共享运行时状态, XML 格式定义如下 (schema namespace urn:d1:state:v1):

<?xml version="1.0" encoding="UTF-8"?>
<d1-state version="1.0" xmlns="urn:d1:state:v1">
  <npus>
    <npu id="npu0" vendor="ascend" model="Ascend 910B" pci-id="0000:01:00.0" status="ACTIVE" hccs-link="up"/>
    <npu id="npu1" vendor="ascend" model="Ascend 910B" pci-id="0000:02:00.0" status="ACTIVE" hccs-link="up"/>
    <npu id="npu2" vendor="ascend" model="Ascend 910B" pci-id="0000:03:00.0" status="ACTIVE" hccs-link="up"/>
  </npus>
  <models>
    <model id="resnet50" path="/opt/dcard/models/resnet50.om" card-id="npu0" loaded-at="2026-07-04T10:00:00Z" total-infer="0" avg-latency-us="0"/>

15.4.2 启动脚本 /etc/d1/activate.sh 示例

v3.2 把原来 v3.1 cann-init.sh 整合到 /etc/d1/activate.sh, 由 systemd 启动 d1-ai-inference.service 或容器 entrypoint 调用:

#!/bin/bash
# /etc/d1/activate.sh - D1 v3.2 AI 推理服务激活脚本
# 由 systemd (d1-ai-inference.service) 或容器 entrypoint 调用
# 退出码: 0=成功, 1=启动失败

set -e

STATE_XML="/var/lib/d1/state.xml"
LOG="/var/log/d1/{ai-inference,health-monitor,cann,acl}.log"
MAX_CARDS=3

mkdir -p /var/lib/d1 /var/log/d1
exec >> "$LOG" 2>&1

echo "=== D1 activate.sh start: $(date -Iseconds) ==="

# (a) 加载驱动
modprobe drv_pcie_ascend || true
modprobe drv_ascend_hal || true

# (b) 扫描 AI 卡 (最多 3 张)
NPU_COUNT=$(npu-smi info -l | grep -c "Ascend 910B" || true)
if [ "$NPU_COUNT" -gt "$MAX_CARDS" ]; then
    echo "WARN: 检测到 $NPU_COUNT 张 NPU, D1 v3.2 限 $MAX_CARDS 张, 取前 $MAX_CARDS 张"
    NPU_COUNT=$MAX_CARDS
fi
if [ "$NPU_COUNT" -lt 1 ]; then
    echo "ERR: 没检测到 NPU"
    exit 1
fi

# (c) 读 XML 持久化 (如果存在)
if [ -f "$STATE_XML" ]; then
    echo "找到 $STATE_XML, 恢复状态"
    # xmllint --xpath "//d1-state:npus/d1-state:npu" $STATE_XML (示意)
fi

# (d) 启动 AI 推理服务 (调用 /usr/local/dcard/bin/ai-inference-service)
/usr/local/dcard/bin/ai-inference-service \
    --state-xml="$STATE_XML" \
    --max-cards="$MAX_CARDS" \
    --vnic-url="vnic://vnic0:50051" \
    --mgmt-url="vnic://vnic0:50053" &
AI_PID=$!
sleep 2

# 验证
if ! kill -0 $AI_PID 2>/dev/null; then
    echo "ERR: AI 推理服务启动失败"
    exit 1
fi

echo "D1 激活成功: $NPU_COUNT 张 NPU, PID=$AI_PID"
exit 0

15.4.3 设计思路 (3 个 why)

  1. 为什么需要启动脚本 (而不是容器 hook): 启动脚本 /etc/d1/activate.sh 在容器 entrypoint 或 systemd 阶段执行, 比 K8s lifecycle hook (postStart / preStop) 更早介入, 能捕获驱动加载失败 / NPU 不可见 / ACL 初始化失败等早期问题, 避免 Pod 起来后再发现推理服务无法启动 (K8s hook 只能回调通知, 不能中止启动). 启动脚本 exit 1 → systemd 不会拉起服务 → kubelet 不会调度 Pod → 失败快速暴露.
  2. 为什么需要 XML 持久化 (而不是纯内存): 重启 AI 推理服务后, 内存状态丢失, 但物理 NPU 上 ACL 上下文 / 模型 .om 加载 / 队列都需要重建, 重建耗时 30s+ (3 张 NPU 加载 3 个 .om 模型). 通过 /var/lib/d1/state.xml 持久化 NPU 状态 + 模型加载情况 + ACL 内存分配 + 队列状态, 重启时按 XML 恢复, 启动时间从 30s+ 缩短到 5s.
  3. 为什么需要 ACL 内存预分配 (而不是动态分配): ACL device 内存 (HBM) 动态分配开销 5-10 ms / 次, 单次推理耗时 2-3 ms, 动态分配占比 200%+, 不可接受. 启动时调 aclInit + aclrtMalloc 按模型预估 (v3.2 限 8192 MB / 模型, 3 卡共 24 GB) 一次性分配, 推理时直接复用, 性能稳定.

15.5 AI 卡状态感知 (Card State Awareness)

AI 卡状态感知 (CardStateAwareness) 监听 Qemu 热插拔 / 热拔插事件, 自动触发注册或去注册, 详见图 9-5 时序图:

  • 事件源: qemu-monitor 命令 device_add / device_del (libvirt API + QMP 协议) (DEVICE_ATTACHED / DEVICE_DETACHED), 通过 QMP (Qemu Machine Protocol) 与 Qemu 通信
  • 事件处理: 使用 libvirt Go binding 的 DomainEventCallback 异步触发 → 调用 §15.4 (e) 注册 或 §15.4 (e) 去注册
  • 重连机制: libvirt connection 异常 (e.g. host reboot) 时异步重连, 丢失的事件依赖下次 dctl reconcile 兜底
  • 下游推送: 状态变化经 HealthService.Watch 推送给所有订阅方 (dctl / 监控)

状态感知触发场景

(1) Qemu guest 启动附加 NPU 设备 → 自动注册到推理服务; (2) NPU 出现 ECC 不可恢复错误 → Qemu 强制 hot-unplug → 状态感知自动去注册; (3) K8s Node 维护 (cordon) → Qemu shutdown → 全部去注册. 这与 §15.4 (e) 的注册与去注册以及 §17 AI 卡部署的 ActivateVM 形成闭环.

  • GetLogs (拉取): 替换 §16.4 HealthService.GetLogs, 按 card_uuid / 时间窗 / 级别 / 源 过滤, 支持 follow_token 续传 (tail 模式). 路径 GET /v1/logs/collector.
  • Watch (流式 tail): server-streaming RPC, 实时推送新增日志行, 类似 tail -F. 路径 GET /v1/logs/collector/watch (gRPC stream). v3.1 留尾巴, v3.2 正式实现.
  • Push (推送到外部系统): 配置 Loki / ES / S3 endpoint, 异步批量推送 (10s 间隔 / 1000 行阈值), 失败本地缓冲 (10MB ring).

15.7.3.5 数据流 (4 源 → 3 类输出)

┌─────────────────────────────────────────────────────────────┐
│  4 类数据源 (tail -F)                                         │
│  ├─ /var/log/d1/ai-inference.log   (JSON, 实时)             │
│  ├─ /var/log/d1/health-monitor.log (JSON, 实时)             │
│  ├─ /var/log/ascend/cann.log       (纯文本, 实时)           │
│  └─ /var/log/ascend/acl.log        (纯文本, 实时)           │
└────────────────────┬────────────────────────────────────────┘
                     │ tail + parse
                     ▼
┌─────────────────────────────────────────────────────────────┐
│  Collector (收集器) + Parser (解析器)                        │
│  → 10MB ring buffer + 内存索引                              │
└────────────────────┬────────────────────────────────────────┘
                     │ 3 类输出
        ┌────────────┼────────────┐
        ▼            ▼            ▼
   GetLogs       Watch         Push
   (拉取)        (流式 tail)    (Loki / ES / S3)
   gRPC unary    gRPC stream   HTTP POST / _bulk

15.7.3 内部组件 (3 大类)

  • Collector (收集器): 4 类数据源 tail + 内存索引 (10MB ring buffer).
  • Parser (解析器): JSON 解析 (ai-inference / health-monitor) + 纯文本正则解析 (cann / acl), 输出统一 LogEntry 结构 (ts / level / source / message / fields).
  • Pusher (推送器): 异步批量推送 (Loki HTTP push API / ES _bulk / S3 PutObject), 失败本地缓冲 (10MB ring).

17AI 卡部署 (DeployService) v3.2 升级

AI 卡部署 (DeployService) 是 v3.2 沿用 v3.1 的独立控制平面, 通过 qemu -netdev user,id=vnic0 -device virtio-net-pci,netdev=vnic0 配置 vNic, 负责 AI 卡 VM 激活 + 部署时告警事件上报. 取代 v3.0/v3.1 的 cann-init.sh 4 阶段脚本 (整合到 §15.4 AI 卡使能模块, 由 /etc/d1/activate.sh 启动脚本触发).

17.1 VM 激活 (ActivateVM)

17.1.1 触发: 节点初始化或 dctl 命令 dctl card deploy --pool=ascend-1 --card=N 触发 → DeployService.

17.1.2 主流程 (5 阶段, 任何一阶段失败立即终止):

  1. (a) dctl 触发: 校验 pool 配额 + 节点资源
  2. (b) Qemu 启动 VM: libvirt domain defineXML + create, PCIe passthrough 绑定 /dev/davinci*
  3. (c) VM 内部装驱动: 通过 cloud-init 或 in-guest agent 执行 A300-9000-npu-driver-{version}.run
  4. (d) 加载固件: ascend-firmware-{version}.run, 与驱动版本必须匹配 (否则报 FIRMWARE_MISMATCH)
  5. (e) 验证 NPU 可见: VM 内 npu-smi info -l 探测, 必须看到期望数量的 NPU (否则报 NPU_INVISIBLE)

17.1.3 成功路径: 调用 §15.4 ActivateCard 完成 AI 卡使能, D-Card Mgr 标记 pool 为 ACTIVE, 触发 HCCS 拓扑完成握手, InferenceService.Init 自动加载 /opt/dcard/models/*.om.

21v3.2 集成场景 (Integration Scenarios) v3.2 新增

v3.2 引入启动脚本 + vNic + XML 持久化后, 典型集成场景包括 3 类:

21.1 场景 A: 全新部署 (Cold Start)

从 0 启动 D1 v3.2 节点:

  1. Kubelet 调度 Pod → 启动容器 entrypoint → 执行 /etc/d1/activate.sh
  2. 激活脚本 (a) 加载驱动 (modprobe drv_pcie_ascend)
  3. 激活脚本 (b) npu-smi info -l → 发现 3 张 NPU → 写 PCI 设备到 /var/lib/d1/state.xml
  4. 激活脚本 (c) XML 不存在 → 创建空 XML, 标记 3 张 NPU = ACTIVE
  5. 激活脚本启动 ai-inference-service → (d) 初始化队列 + (e) ACL 内存分配 (24 GB) + (f) 加载 3 个 .om 模型 (resnet50 / bert-base / llama-7b) + (g) 启动 vNic 监听 (vnic://vnic0:50051 + vnic://vnic0:50053)
  6. kubelet readiness probe 调 HealthService.Check → 返回 SERVING → Pod Ready

21.2 场景 B: 已有部署重启 (Warm Start, 状态恢复)

已有 D1 v3.2 节点, AI 推理服务重启 (e.g. systemd restart):

  1. systemd 启动 d1-ai-inference.service → 执行 /etc/d1/activate.sh
  2. 激活脚本 (a) 加载驱动
  3. 激活脚本 (b) npu-smi info -l → 发现 3 张 NPU (PCI-id 与上次一致)
  4. 激活脚本 (c) XML /var/lib/d1/state.xml 存在 → XMLPersistence.Load() → 恢复 NPU 状态 + 模型路径 + ACL 内存分配 + 队列状态
  5. ai-inference-service 启动 → (d-f) 跳过模型重载, 直接复用 XML 中的 ACL context (节省 25s+)
  6. 激活脚本 (g) vNic 监听启动

关键收益: 冷启动 30s+ → 热启动 5s (节省 25s).

21.3 场景 C: qemu 热拔插 1 张 NPU (Dynamic Reconfigure)

运维通过 qemu-monitor 拔出 1 张 NPU:

  1. 运维执行 (qemu) device_del id=npu0
  2. QMP 协议推送 DEVICE_DETACHED 事件 → libvirt Go binding DomainEventCallback 触发
  3. §15.5 状态感知模块接收事件 → 解析 card_uuid=npu0
  4. §15.4 (g) 反向调用 UnregisterCard → 标记 npu0 = CARD_UNREGISTERED
  5. 调度器从 routing_table 中剔除 npu0 → 模型路由回退到 2 个池子 (npu1 + npu2)
  6. ai-inference-service 内部卸载 npu0 的 3 个 .om 模型 (resnet50 / bert-base / llama-7b) + 释放 ACL 内存 (8192 MB)
  7. HealthService.Watch 流推送 CARD_REMOVED 事件 → dctl / 监控收到
  8. 报告 HealthService.ReportAlert (REGISTER_FAIL 子码, level=WARN)

反向场景 (热插入 1 张 NPU): (qemu) device_add driver=vfio-pci,host=0000:04:00.0 → 触发加载模型 + 加入 routing_table → 3 池恢复.

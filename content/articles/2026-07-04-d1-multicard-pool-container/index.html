1概述

D-Series AI Card D1 是面向 华为昇腾 Ascend 910B + CANN 8.0.RC2 + HCCS 的多 AI 卡池化与容器化子系统. 核心目标: 把 3 张物理 NPU 抽象成 3 个独立推理池 (每池加载同一套模型), 通过 vNic URL 寻址对外提供多模型推理服务, 支持热拔插加减池 + 多节点故障倒换.

17 个内部 API URL · 3 个子系统 + 2 个扩展子系统 · 3 张 NPU (1-3 池, 可扩展到 16 节点) · 22 个错误码

核心 KPI

3独立推理池
5推理模型
17内部 API URL
22错误码
0外部暴露端口

📌 关键不变量

  • 零外部端口: 所有 API 走 Unix Domain Socket (unix:///var/run/d1/*.sock) 或 vNic URL (vnic://vnic0:5005x), 不对外暴露任何 TCP 端口.
  • 探测走 HTTP, 变更走 gRPC: 健康/状态/日志拉取类用 HTTP GET 5s 轮询 (无长连接), 推理/激活/重定向类用 gRPC Unary (高性能, 大流量).
  • 异步优先: AI 卡激活、日志打包、跨节点重定向均为异步任务 (返回 task_id, 客户端轮询).
  • 模型同构: 所有 AI 卡加载相同 3 模型副本, 故障时无需重新加载, 直接路由.

3设计思路

3.1 设计目标

  • 池化: N 张 AI 卡 → N 个独立推理池, 每池加载相同模型副本, 客户端按 model_id 路由.
  • 容器化: AI 推理服务跑在 VM 容器内 (qemu + virtio-net), AI 卡以 PCIe passthrough 暴露.
  • 热拔插: AI 卡可热拔插 (qemu-monitor device_add/device_del), 运行时加减池.
  • 故障倒换: 单 AI 卡故障由本节点 LB 重分配 (3s); 整节点失败由跨节点轮询 + 重定向 (16s).
  • 零外部依赖: 不依赖外部 K8s Operator / Prometheus / Loki, 自洽.

3.2 3 层架构

层职责关键组件
硬件层PCIe + HCCS + HBMAscend 910B NPU × 3 + drv_pcie_ascend 驱动
运行时层CANN ACL 框架 + ACL 内存 + 推理队列CANN 8.0.RC2 + ACL + HCCS peer
服务层vNic URL 寻址 + 调度 + 健康 + 日志activate.sh + state.xml + LogCollector + vNicServer

3.3 双协议策略

v3.2.1 协议修订: 探测类走 HTTP GET 5s 轮询 (无长连接, 故障恢复 < 1s), 变更/推理类保留 gRPC Unary (高性能). 客户端只调 vNic URL 或 UDS, 不感知底层协议.

类别主路径备路径典型场景
健康轮询GET vnic://vnic0:50053/health/check?scope=...—5s 轮询 liveness / readiness
状态查询GET vnic://vnic0:50053/health/poll?scope=...&since=...—5s 轮询状态变化
日志查询GET vnic://vnic0:50053/logs?from=...&to=...—日志拉取 + tail

3.4 明确不做 (Out of Scope)

  • Pool Manager (D2): 跨集群池管理
  • Hot-Swap 故障自愈 (D3): 自动重启失败 NPU
  • Federation (D4): 跨 DC 联邦推理
  • Edge (D5): 边缘节点
  • 推理高级调度: A/B 测试 / 蓝绿 / QPS 精细路由 (v3.2 只做 model_id 路由 + round-robin/least-conn)

4核心流程

本章列出 5 个核心流程, 后续 §5-§9 子系统章节只讲组件 + 接口, 不再重复流程. 所有流程均使用 vNic URL 寻址, 异步任务返回 task_id, 客户端通过 HTTP GET 轮询进度.

4.1 AI 卡激活 (1 个 URL 异步, 7 阶段 a→g)

入口: 周边模块 (dctl / Omtask / SwitchController) 调用 1 个 URL 启动 AI 卡激活, 异步 (典型 15-30s, 跟启动脚本 7 阶段一一对应).

POST vnic://vnic0:50053/aicard/activate      (HTTP/JSON 主路径, 异步, 返回 task_id < 100ms)
等价 gRPC: AICardManager.ActivateCard (Unary)

Body (JSON):
{
  "card_uuids": ["npu-0", "npu-1", "npu-2"],   // 可选, 省略 = 当前 PCIe 扫到的全部
  "max_wait_s": 600,                            // 最大等待时间
  "xml_path": "/var/lib/d1/state.xml"          // 持久化路径 (可选, 默认)
}

→ 202 Accepted
{
  "task_id":     "act-2026-07-08-001",
  "started_at":  "2026-07-08T03:00:00Z",
  "total_phases": 7,
  "status_url":   "vnic://vnic0:50053/aicard/activate/act-2026-07-08-001/status"
}

7 阶段 (后台异步执行, 客户端通过 status_url 轮询):

阶段动作耗时幂等
a. 驱动加载modprobe drv_pcie_ascend 加载 PCIe 驱动2-3s是
b. 设备扫描npu-smi info -l 扫到所有 AI 卡, 写 PCI BDF + UUID 到 state.xml< 1s是
c. XML 持久化读 /var/lib/d1/state.xml 快照 (存在则恢复状态), 写新 XML 临时文件 → fsync → rename< 50ms是
d. 队列初始化3 个推理队列创建 (每池 1 个), round-robin 绑定< 1s是
e. ACL 内存分配每卡分配 24 GB HBM (3 卡 × 8GB 独占 + 共享)2-5s否 (失败回滚)
f. 模型加载每池加载同一套 3 模型 (resnet50-io / bert-base-file / llama-7b-context), .om 格式5-15s否 (失败回滚)
g. vNic 监听启动 vNicServer 监听 :50051 (推理) + :50053 (管理), virtio-net 后端< 1s是

状态机 (a→b→c→d→e→f→g → DONE, 任一阶段失败 → FAILED + 回滚):

4.2 AI 推理调用 (3 卡 3 池, model_id 路由)

POST vnic://vnic0:50051/infer    (HTTP/JSON 主路径)
等价 gRPC: AIInferenceService.Infer (Unary, 备路径)

Body:
{
  "model_id": "resnet50-io",
  "features": [0.12, 0.85, ...]   // 或 bytes (模型特定)
}

→ 200 OK
{
  "task_id": "inf-001",
  "result": "ransomware_detected"  // 或 scores/embeddings
  "latency_ms": 18
}

路由策略:

  1. 客户端传 model_id, 调度器查 routing_table (model_id → pool_id 列表)
  2. 从候选池中选 1 个: round-robin (默认) / least-conn (负载均衡)
  3. 热拔插时, 路由表自动剔除失效池

4.3 健康轮询 (HTTP GET 5s, since 增量)

// 单次健康检查
GET vnic://vnic0:50053/health/check?scope=full
→ {"healthy": true, "level": "GREEN", "metrics": {"temp": 42.5}}

// 状态感知 (since 增量, 替代旧 gRPC Watch stream)
GET vnic://vnic0:50053/health/poll?scope=POOL&since=1717700000
→ {"events": [{"type": "CARD_REMOVED", "ts": 1717700005}], "next_since": 1717700005}

3 层检查: liveness (进程 + vNic 200) / readiness (驱动 + ACL 24GB + AI 卡) / model-ready (3 模型全 loaded). 切换阈值: health_score ≥ 0.85 + 3 层全 PASS + 连续 3 次 5s 轮询 (15s) 全 PASS.

4.4 日志打包 (XML 注册 + 异步 trigger/query/download)

XML 配置 (/etc/d1/log-collector.xml) 维护日志路径, 每个 <node> 含 filename + src 2 个字段. 周边模块通过 API 动态注册. filename = "*" 表示打包整个目录所有文件.

// 1. 注册路径
POST vnic://vnic0:50053/logs/registry/node   (HTTP POST)
Body: {"filename": "ai-inference.log", "src": "/var/log/d1/ai-inference.log"}
→ {"accepted": true, "node_id": "node-007", "total_nodes": 12}

// 2. 触发打包 (异步, < 100ms 返回 task_id)
POST vnic://vnic0:50053/logs/package/trigger
→ 202 {"task_id": "pkg-2026-07-08-001", "estimated_size_mb": 850, "eta_s": 180}

// 3. 查询进度 (HTTP GET 5s 轮询)
GET vnic://vnic0:50053/logs/package/{task_id}/status
→ {"state": "PACKAGING", "percent": 67, "current_node": "acl.log", "eta_s": 60}

// 4. 下载 (HTTP Range 断点续传)
GET vnic://vnic0:50053/logs/package/{task_id}/download
→ .tar.gz (7 天 TTL 自动清理)

4.5 故障倒换 (本控 LB / 跨节点重定向)

单 AI 卡故障 (e.g. XID 79): 本控 LB 自动剔除 (3s SLA), 流量切到剩余 2 池.

整节点故障 (3/3 AI 卡全 fail 或节点 off): 跨节点轮询 etcd (16s SLA), 找 health_score 最高的 active 节点重定向. 16 节点集群以 4 节点举例.

// 跨节点重定向 (客户端调用, 内部自动选节点)
POST vnic://vnic0:50053/peers/redirect
Body: {"idempotency_key": "uuid", "request_payload": {...}}
→ {"new_vnic_url": "vnic://vnic1:50051", "response_payload": {...}}

完整故障倒换逻辑见 §9.

5AI 卡管理子系统

负责 AI 卡生命周期管理 (创建池 / 激活 / 去注册 / 状态感知).

5.1 核心组件

  • AICardManager: 创建/销毁 Pool, 注册/去注册 AI 卡, AI 卡 7 阶段激活 (后台异步)
  • StatePersistence: XML 持久化 (/var/lib/d1/state.xml), 启动时 Load(), 变更时原子 rename
  • CardStateAwareness: qemu-monitor QMP 事件订阅 (DEVICE_DETACHED/ATTACHED), 5s 轮询 vNic URL 出口

5.2 Pool 创建流程 (8.2 旧)

  1. AICardManager.CreatePool (POST /aicard/pools): 选 3 张卡 → 创建 Pool → 加载模型 → 加入 routing_table
  2. state.xml 记录 (pool_id, card_uuids, model_ids, acl_mb, queue_size)

5.3 AI 卡去注册 (热拔插)

  1. qemu-monitor device_del id=npu-0 → QMP DEVICE_DETACHED 事件
  2. CardStateAwareness 写入 AI 卡状态寄存器 (共享内存)
  3. 5s 轮询 GET vnic://vnic0:50053/health/poll?since={ts} 感知
  4. 调度器从 routing_table 剔除 npu-0
  5. 异步卸载 npu-0 上的模型 + 释放 ACL 内存 (8GB)

完整 API 表 见 §10. 复用核心流程 见 §4.1 (激活) / §4.5 (故障倒换).

6.1 VNicServer (vNic 通信)

2 个端口分离: 业务面 :50051 (推理, 高频) + 管理面 :50053 (运维, 低频). 鉴权: mTLS 双向, 业务证书 vs 运维证书. 详细 API 见 §10.2.

6.2 MultiModelInference (多模型推理)

3 池模型同构 (每池 3 模型 resnet50-io / bert-base-file / llama-7b-context). 调度: model_id 路由 → round-robin (默认) / least-conn. 热拔插时 routing_table 自动剔除. 完整推理时序 见 §4.2.

6.3 HealthMonitor (健康监控)

3 层检查 (liveness / readiness / model-ready), 1s 周期后台采样, 通过 §4.3 HTTP GET 5s 轮询出口. 不做日志收集 (v3.2.2 拆到 §6.4 LogCollector).

6.4 LogCollector (日志收集, v3.2.2 重构)

XML 配置中心 + 异步打包. 9 个 API: 4 注册 (POST/DELETE/GET/Watch) + 5 打包 (Trigger/Status/Download/Cancel/Delete). 详细见 §4.4.

POST vnic://vnic0:50053/omtask/deploy    (HTTP/JSON 异步, 替代旧 dctl CLI 命令)
Body: {"upgrade_version": "CANN-8.0.RC2", "dry_run": false, "rollback_on_fail": true}

8防勒索切换子系统

容器外防勒索服务 (CPU CNN/RNN 推理) 检测到勒索后, 一旦 AI 卡激活 + 健康稳定, 自动切换到 AI 推理 (15 倍加速, 误报率 1.2% → 0.3%).

8.1 组件

  • anti-ransom (容器外 CPU 进程): IO 特征提取 → CNN/RNN 推理 → 输出判定
  • SwitchController (sidecar): 监听健康 (5s 轮询 §4.3) → 触发切换阈值 (≥ 0.85 + 3 层 PASS + 15s 稳定) → 通知 MessageRouter
  • MessageRouter: 切换后 IO 消息路由 cpu_inference → ai_inference

8.2 切换状态机 (5 态)

状态说明
CPU_ONLY初始, CPU 推理
HYBRID过渡, CPU + AI 并行 (2s 缓冲)
AI_ONLY终态, AI 推理
CPU_FAILCPU 故障, 自动重启
AI_FAILAI 故障, 回退 CPU

8.3 入口: 1 个 URL

POST vnic://vnic0:50053/switch/to-ai   (HTTP/JSON, sidecar 内部)
Body: {"trigger": "HEALTH_OK", "buffer_seconds": 2}
→ 200 {"accepted": true, "eta_ms": 100}

切换流程 详见 v3.2 §24.3 活动图 / 时序图 (3 张 SVG). 5 阶段: ① 触发 → ② 健康探测 (3 层 + 15s 稳定) → ③ 模型就绪 → ④ 切换决策 → ⑤ 路由生效 + CPU 关停. 总耗时 ~26s.

9.2 单 AI 卡故障 → 本控 LB (3s)

  1. N3-卡-0 PCIe AER XID 79 → HealthService 标记 UNHEALTHY
  2. N3 LB routing_table 剔除 N3-卡-0 (3s)
  3. resnet50 流量切到 N3-卡-1/2 (round-robin)

9.3 整节点故障 → 跨节点重定向 (16s)

  1. N4 心跳丢失 ×3 → etcd 标记 NODE_DOWN (10s TTL)
  2. 其他节点 Watch etcd 收到事件
  3. 客户端原 N4 路由 → 调 POST /peers/redirect
  4. 选 health_score 最高的 active 节点 (N1 = 0.92) 重定向
  5. etcd Watch 重新注册 (节点恢复后)

9.4 入口: 1 个 URL

POST vnic://vnic0:50053/peers/redirect   (HTTP/JSON)
Body: {"idempotency_key": "uuid", "request_payload": {...}}
→ {"new_vnic_url": "vnic://vnic1:50051", "response_payload": {...}}

节点状态机 (4 态 HEALTHY/DEGRADED/OFFLINE/FAILED, 8 迁移条件) 见 v3.2 §25.5.

10统一 API 设计

📐 设计原则

  • URL 寻址: 所有 API 通过 vNic URL (vnic://vnic0:5005x/...) 或 UDS (unix:///var/run/d1/*.sock) 暴露, 不对外暴露 TCP 端口.
  • HTTP/JSON 主路径 + gRPC Unary 备路径: 探测类只支持 HTTP, 变更/推理类 HTTP + gRPC 双协议.
  • 最小化: 17 个内部 URL, 涵盖 5 个子系统 + 1 个健康服务, 无冗余.
  • 异步优先: AI 卡激活 / 日志打包 / 跨节点重定向 返回 task_id, 客户端轮询.

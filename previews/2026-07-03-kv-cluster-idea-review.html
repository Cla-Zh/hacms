2.3 业界核心共识 (一句话)

总结

业界对 "分布式 KV 池" 的实现已收敛成清晰模式: HBM → DRAM → (本地 NVMe || RDMA-attached 远端 NVMe) → 共享存储 5 层体系。 RDMA 替代本地 SSD 是显式设计目标, 不是反模式, 但前提是 ≥200 Gbps + sub-ms RTT + GPUDirect end-to-end。TCP-only 路径反而输给本地 NVMe 5-10× (NVIDIA Dynamo 文档实测)。

4128 节点算不算 "超大"? 业界规模盘点 (从 agent3)

4.1 KV 集群规模业界对照

图 1. 业界 KV 集群规模对照 (对数横轴)。用户提出的 128 节点在 Moonshot Mooncake "数千节点" 面前 属于"中型", 不算"超大"。但 Moonshot 公开数字是唯一直白说法, 其它多数头部厂商 (OpenAI/Anthropic/Alibaba) 不披露具体节点数, 因此真实工业级头部集群规模可能远超 10k+ 节点。

4.2 一致性挑战 (业界普遍痛点)

  • 强一致 vs 多副本: Mooncake Store 提供 Put 后强一致, 但多副本是 best-effort, 无 Raft/Paxos。
  • 缓存命中与冷启动: Mooncake 报告 59%~498% 容量提升, 但 LMCache 论文明确指出 "context truncation 可让 prefix cache hit rate 减半"。
  • 网络分区: Mooncake TE 自带 multi-NIC 聚合 + 自动 failover, etcd 元数据层耐链路抖动。
  • 5 层容量规划: NVIDIA Dynamo G1 HBM / G2 DRAM / G3 NVMe / G3.5 CMX / G4 企业存储, 每层延迟和成本不同。
  • 淘汰策略: SGLang Radix + LRU / vLLM block-LRU / PagedAttention 把碎片从 60-80% 降到 < 4%。

4.3 128 节点规模 — 客观结论

诚实判断

128 节点 ≠ 超大。Mooncake 数千节点; DeepSeek 3FS 公开 180 存储 + 500 客户端; 头部厂商 OpenAI/Anthropic/Alibaba/Baseten 不公开具体节点数, 但从公开 TTFT 基准推断规模显著大于 128 节点。
但 "128 节点" 仍需要至少 IB NDR 400 Gb/s 或 Spectrum-X 每节点 + Mooncake TE/NIXL 数据通路 + Mooncake Store/LMCache 缓存层 + Dynamo/Kthena 路由。 硬件配置需求接近"大型生产" 等级, 不应低估复杂度。

维度事实结论

5.2 假设 2 — 8-12 张 NPU 做投机 / 哈希 / 路由 错

维度事实问题
昇腾 310P 定位 176 TOPS INT8 / 70 TFLOPS FP16 / 48 GB LPDDR4X / 204 GB/s 带宽 边缘推理芯片, 不是数据中心推理主力
vs H100 对比 算力 ~7%, 带宽 ~6% 跑 LLM decode 反规格使用
KV 路由延迟 NPU 调用链 Host→CANN→AICore 单次 50-200μs; host CPU + radix tree <10μs 业界无 NPU 做 KV 路由的先例

loop1 判决: 错 — 310P 用 LPDDR4X 而非 HBM, KV 频繁存取立即成瓶颈。vLLM/SGLang/llm-d 全部用 host CPU + radix tree 做前缀匹配, 延迟 <10μs。"NPU 做 KV 哈希路由"无任何业界先例。

5.3 假设 3 — 不下盘直接转发 半对

业界方案实现路径延迟
NIXL Prefill GPU VRAM → NIXL → (RDMA/NVLink) → Decode GPU VRAM, 不落 host DRAM/NVMe 5-15μs GPU↔GPU
Mooncake TE RDMA + GPUDirect, zero-copy VRAM↔VRAM / DRAM↔VRAM / NVMe-oF↔DRAM/VRAM 87-190 GB/s (4×200G / 8×400G RoCE)
LMCache NIXL + RDMA / NVLink / TCP, 不落 SSD —
FlowKV (阿里) 减少 kernel 调用次数 KV 传输延迟 0.944s → 0.053s (↓96%)

loop1 判决: 半对 — "不下盘" 业界确实做, 但实现方式是 GPU↔GPU via RDMA + NIXL/Mooncake TE, 一跳 5-15μs。用户想象 "NPU 中继转发" 是用户发明环节, 没有先例。NPU 中继三跳后总延迟 50-100μs, 比 RDMA 直传慢一个数量级。

5.4 假设 4 — 跨节点 NPU 叠加 KV 错

相关技术实际是什么用户混淆了什么
CacheBlend (EuroSys'25) 软件层 selective recompute: 从多个 chunk 预计算 KV 中选 15% 重新算, 复用 85%。 不是 NPU 算子融合, 是软件层重算
KV-Cache Fusion block-level fusion / adaptive token merging / layer-wise sharing / low-rank projections, 主做压缩。 是压缩, 不是 "不同来源 KV 相加"
Tensor Parallel (TP) 不同 GPU 持有 KV 的不同 head 切片, attention 计算时 all-reduce 合并。 算法层合并, 不是 "NPU 取出 KV 再叠加再分发"
Mooncake KV pool KV 散布在多节点, 按需拉取完整 chunk。 不 "叠加"

loop1 判决: 错 — 跨节点 KV 融合 没有任何业界先例。用户混淆了 3 件事: (a) prefix cache reuse — 整段前缀重用, KV 不叠加; (b) CacheBlend — 重算 15% 修补, 不叠加; (c) Tensor Parallel — 同一层不同 head 切片, 算法层合并, 不是 KV 数据层叠加。 更关键: KV 张量数学上不可简单相加 (位置编码不同, 语义不同), 用户描述的抽象本身不成立。

5.5 假设 5 — 网络带宽打本地 SSD 反模式

反模式 — 方向搞反了

业界共识 反过来: 用网络带宽 绕过本地 SSD, 远端 RDMA-enabled SSD → GPU VRAM (Mooncake/VAST/LMCache 一致做法)。本地 NVMe 反而成为瓶颈:

  • VAST Data 原文: "Local NVMe works as a spill tier, but a few standalone drives top out well below what modern GPU pools can consume, so it becomes a throughput limiter, while also consuming scarce PCIe bandwidth and CPU cycles."
  • Mooncake: "constructing a multi-level caching pool on high-speed interconnected DRAM/SSD resources" — SSD 是远端 pooled storage, 不是本地落盘。
  • 延迟叠加: 用户路径 网络→本地 DRAM→本地 SSD→本地 DRAM→GPU = 3-5 跳, 每跳 10-50μs, 比 RDMA 直传 GPU VRAM 慢 30-100μs。
  • 寿命问题: NVMe 3 DWPD (enterprise 顶级) 也扛不住 KV cache 频繁写, 几个月就要换盘。
  • 带宽浪费: 本地 NVMe 写入 (7-14 GB/s) 比 RDMA (50 GB/s) 慢 3-7 倍, 反而拖垮整体吞吐。
  • 无容量收益: 数据本来就在远端 SSD, 再写一份到本地 SSD, 总容量没增加, 只增加延迟和成本。

loop1 判决: 明确的反模式 — 业界路径是 Remote Disaggregated Storage Pool (RDMA + NVMe-oF + GPUDirect Storage) → GPU VRAM, 完全绕过本地 NVMe。

7我的最终判断 + 5 条下一步建议

7.1 最终判断

最终结论

用户的核心 insight 是对的: 用网络扩展 KV 容量、减少重算, 是业界 (Mooncake/LMCache/DeepSeek 3FS) 已验证的方向。
但 5 条具体路径中 3 条错 (含 1 条反模式), 2 条部分对, 没有一条"完全对"。最大问题不是思路而是 硬件定位: 8-12 × 昇腾 310P 是边缘推理芯片, 不适合做数据中心 LLM 推理主力 / KV 路由 / KV 转发中继。

假设评级一句话
H1 硬件足够部分对单点 OK, 但 KV 容量爆炸被严重低估; 需要 RDMA NICs 明确列规格
H2 8-12 × 310P错310P = 边缘芯片, 算力 ~7% H100; KV 路由无 NPU 先例
H3 不下盘转发半对业界做法 GPU↔GPU via RDMA, 不应中继 NPU
H4 KV 融合错KV 数学上不可简单相加; CacheBlend 是软件层重算
H5 网络打 SSD反模式方向反了 — 应是远端 RDMA-SSD → GPU VRAM

7.2 5 条下一步建议 (按优先级)

建议 1 — 重新评估硬件选型 P0

不要把 8-12 × 310P 当主推理。建议如下分层:

  • 主推理: 用 Ascend 910B/C (或 NVIDIA H100/B200), 跑 LLM decode + prefill。
  • Draft 模型: 1-4 张 310P, 每张跑独立 1-3B EAGLE draft (算力需求 1/10~1/70 主模型)。
  • 预处理: 4-8 张 310P, 跑 vision encoder / audio tokenizer / H.265 解码。
  • KV 量化压缩: 1-2 张 310P, KVQuant/KIVI 在线量化。
  • 网络 IO: 引入 DPU/SmartNIC (NVIDIA BlueField-4 / 华为 HCCS / ConnectX-7), 不要让 NPU 充当 IO 中继。

建议 2 — 采用 Mooncake TE + NIXL 做 KV 转发, 删除"NPU 中继"环节 P0

架构直接对齐业界 reference: Prefill GPU → RDMA zero-copy (Mooncake TE / NIXL) → Decode GPU VRAM, 一跳 5-15μs。中继 NPU 是用户发明环节, 没有先例, 多 50-100μs 延迟。

  • vLLM-Ascend 集成 Mooncake Connector: vLLM-Ascend PD Disagg
  • 如果非国产栈: NVIDIA NIXL + Dynamo KVBM

建议 3 — KV "复用" 用 prefix cache + CacheBlend, 不做"NPU 叠加" P1

用户的 H4 在数学上不可能。正确做法:

  • prefix cache: SHA-256 of token sequence, 整段前缀重用, KV 不叠加。LMCache / vLLM / SGLang 全都这样做。
  • CacheBlend (EuroSys'25): 选 15% token 重算修补, 复用 85%, 软件层操作, 不需要新 NPU 算子。
  • 量化压缩: KVQuant (3-bit perplexity 退化 <0.1) 或 KIVI (2-bit), 解决容量爆炸。

建议 4 — 把"本地 SSD"从 KV hot tier 中移除 P1

业界路径是反过来的: 远端 RDMA-enabled SSD pool → GPU VRAM。具体方案:

  • 存储层: DeepSeek 3FS 模式 (180 节点 × 16×14TB NVMe, RDMA) 或商业 VAST DASE / WEKA Augmented Memory Grid / IBM Storage Scale ECE。
  • 数据通路: Mooncake TE / NIXL (含 GPUDirect Storage backend)。
  • 淘汰策略: Mooncake "soft pin / hard pin" + LRU; SGLang RadixAttention + LRU。
  • 本地 NVMe 仅作为 spill tier, 不参与 KV hot path。

建议 5 — 立即可落地的 3 个 PoC P2

基于以上修正, 优先做:

  1. PoC-1: vLLM-Ascend + Mooncake TE PD Disaggregation (1-2 周可成), 验证 KV zero-copy 跨节点转发。基线 5-15μs GPU↔GPU 延迟。
  2. PoC-2: 8-12 × 310P 跑 EAGLE-3 draft (2-3 周), 验证 2-6× decode 加速比。目标: draft 算力需求 1/10~1/70 主模型, 1-2 张 310P 即可服务 1 张 910B。
  3. PoC-3: 128 节点 5 层存储 (HBM→DRAM→远端 NVMe→CMX→共享) (4-6 周), 用 LMCache + NIXL + GPUDirect Storage, 验证 RDMA 替代本地 NVMe 的延迟/带宽假设。

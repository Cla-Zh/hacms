关键洞察: H5 假设「网络打本地 SSD」是 反模式 — 本地 NVMe 单跳延迟 (80μs) 比 RDMA-远端-NVMe (5μs) 还慢 16×. 业界主流路径是 远端 RDMA-SSD → GPU VRAM (Mooncake / GPUDirect Storage), 不再把 KV 锁在本地盘. 推论: KV 集群的存储拓扑应以 RDMA fabric 为中心 重新设计, 本地 NVMe 仅作 L0 写入缓冲.

3PoC-3 · KV cache 数学不可叠加 + 容量爆炸

三层证据否定「KV tensor 线性求和」方案: RoPE 位置编码 + softmax 非线性 + 多 head 不一致. 同时给出 5 个模型在 128k 上下文下的单实例 KV 容量.

数学证明 · 3 个不可叠加的根因

尝试: Kmerged[i] = KB1[i] + KB2[i] (假设 B1, B2 共享 prefix) 问题 1 · RoPE 位置编码: Ki = RoPE(xi, posi) ⇒ KB1[i] 和 KB2[i] 来自不同 绝对/相对位置 → KB1[i] + KB2[i] 在物理上 无意义 问题 2 · Attention 非线性: Attn(Q, K) = softmax(QKT / √d) · V softmax(·) 不可线性分解, 因此 sum-of-K 不等于 sum-of-attn 问题 3 · Multi-head 不一致: 不同 head 在不同 batch 上学到不同 attention pattern → 跨 batch 求 K 缺乏一致坐标系 结论: 业界走 prefix matching + CacheBlend (15% 重导), 不做 tensor 求和.

关键洞察: H4 假设「KV 跨请求叠加」数学上 不可行 — RoPE 让 K 携带位置信息, softmax 阻断线性分解, 多 head 模式不一致. 即使绕开数学, 单实例 128k 容量 也很恐怖: Llama-3-70B = 43 GB (≈ 半个 H100 80GB HBM), 405B = 68 GB (单卡 HBM 装不下), DeepSeek-V3 = 524 GB (要 7 张 H100 才装得下). 推论: KV 集群必须走 prefix matching + 压缩 (KVQuant) + 分层存储 (HBM→DRAM→RDMA-NVMe→S3) 路线, 而不是叠 K 张量.

🟢 关键洞察 — AI 数据平台的本质
业界 AI 数据平台不是"另存一份数据", 而是 3 层价值:
  1. 数据治理: 版本控制 (DVC/LakeFS/Pachyderm) + 数据血缘 (OpenLineage/Marquez) + 质量评分
  2. 数据流水线: 标注 / 去重 / 切分 / 格式转换 (Parquet/Arrow/Safetensors) / 向量化 (embedding-on-write)
  3. Agent 数据通道: 长期记忆 / 工具调用日志 / 多模态素材库 / RAG 向量库
存储的核心价值 不是"放数据", 而是 让 AI 可信、可复现、可治理. NVIDIA AIDP 把 GPU 嵌入数据通路是行业大方向.

1.2 Hugging Face Xet — 一个值得借鉴的细节

  • 问题: Git LFS 修改几 MB 要重新上传整个 GB 文件 (LFS 是文件级)
  • Xet 解决: 把文件切成 ~64KB chunks, 装进 xorb 容器, chunk-level 去重, 修改只重传变化的 chunk
  • 多 HTTP 并行: 不再串行单连接
  • ML 原生格式: Safetensors / Parquet / Arrow, 适配训练场景
  • 用户可借鉴: 如果做本地 SSD 数据湖, 可以做 chunk-level 去重, 大幅降低训练数据集存储成本

2方向 2 — 存算分离的 7 个硬优势 (技术上只有分离能做到)

业界说"存算分离省钱"是软优势. 我梳理了 7 个真正技术上的硬优势:

#硬优势技术解释业界方案
1跨集群共享 checkpoint训练集群断点保存到共享存储, 推理/微调集群可即时加载WekaFS, VAST Data, NetApp ONTAP AI
2数据集快照 (reproducibility)训练数据版本化, 任何时刻可恢复"那一刻的数据"LakeFS, DVC, Pachyderm
3多租户隔离一个存储服务多个训练/推理任务, 性能隔离 + 安全隔离NetApp FlexGroup, VAST DataEngine

5大模型权重分发训练完 700GB 权重从存储拉满 8 卡, 不走网络慢路径NVIDIA Magnum IO, NCCL
6跨集群 Federated Learning多个数据中心联合训练, 不集中数据只共享梯度ByteDance FedML, Flower

🟠 反例 — 存算分离不是万能的
Glenn Lockwood (前 Uber, 现 NVIDIA 顾问) 写过一个反例: 存算一体 (H100 + 本地 NVMe) 在某些 RLHF / inference serving 场景反而更快, 因为:
  • 本地 NVMe 延迟 < 100µs, 而 NFS 延迟 1-10ms (差 10-100x)
  • RDMA 带宽可以饱和 (Mooncake 87 GB/s), 但 CPU 中断开销吃掉 30%
  • 推理 KV cache 频繁随机读, 本地 NVMe 比 NFS 快
所以 不是"存算分离 vs 一体" 的二选一, 是 数据准备阶段用分离, 推理 serving 阶段用一体.

📌 关键洞察 — 6 个阶段的 IO 模式完全不同
没有"一个存储解决所有阶段"的事. 业界共识是 分层 + 多套存储:
  • 数据准备: 对象存储 (S3) + Spark/Fluid 调度
  • 训练: 高带宽并行文件系统 (WekaFS / VAST / DDN) + GPUDirect
  • 推理: 跨节点 KV 集群 (Mooncake) + 本地 NVMe + HBM
  • RLHF: 高 IOPS 共享文件系统 (Anyscale/Modal) + 沙箱 root
这就是 NVIDIA Dynamo 定义的 G1-G4 层级 (HBM → GPU 本地 DRAM → 节点 NVMe → 远端 RDMA-SSD).

🟣 Agent 时代的关键变化
  1. 存储不再是"训练数据", 而是 "agent 的运行环境". 一个 agent 启动时需要: 长期记忆 (历史) + 工具日志 (历史) + 知识库 (RAG) + 多模态素材. 全部要在毫秒级拉到 sandbox.
  2. 存储的语义从"被动" 变成 "主动": 存储要知道 agent 在干什么 (a) 主动预取 agent 下一步可能用的工具/记忆 (b) 自动归档旧会话 (c) 实时同步跨 agent 状态
  3. EU AI Act (2026-08 生效) 要求 AI 系统保留训练/推理的"输入-输出-决策"可追溯日志, 存储的合规性要求大幅提升

4.2 业界领先方案 — Pinecone vs Weaviate vs Milvus

  • Pinecone: Serverless, 无运维, 按 query 付费. 适合不想管基础设施的团队.
  • Weaviate: 开源, 自带 hybrid search (BM25 + vector), 模块化
  • Milvus: 开源, 大规模 (亿级向量), GPU 加速 (CAGRA)
  • LanceDB: 嵌入式 (类似 SQLite), 适合本地 RAG

5方向 5 — 用户硬件的 7 个真实差异化机会

这是报告核心. 用户硬件组合: 256 节点 × (节点内 NPU 昇腾 310P 类 + 极致 NVMe SSD + 高带宽网络). 必须避开 Mooncake/LMCache/InfiniStore 跨节点 KV 赛道, 那是 LLM serving 的事, 跟用户硬件错位.

5.1 用户硬件的 4 个独特优势

  1. 256 节点规模 vs WekaFS/VAST 千节点集群: 用户可做 单集群 in-node 优化, 别人要做跨集群协议
  2. 节点内 NPU: 节点内有专用 AI 算力, 不只是通用 CPU 做数据预处理
  3. 极致 SSD: 节点内可以做 in-line compression / dedup / indexing, 不增加网络跳数
  4. NPU + SSD 紧耦合: 可做 DPU-less 的数据处理 (e.g. embedding 生成直接在 NPU, 数据落 SSD 时已经向量化)

5.2 7 个差异化机会 (DIFF-1 ~ DIFF-7)

🥇 DIFF-1: NPU 原生向量化 / Embedding-on-Write

可行性: 高 vs VAST InsightEngine (用 GPU) vs Pinecone (serverless 单独跑)

用户做什么: 数据落 SSD 那一刻, NPU 已经生成 embedding + 索引 + chunk dedup 标记. 别人要么用 GPU (贵), 要么用 CPU (慢). 用户用 NPU 做 embedding-on-write 是天然差异化.

算力验证: Ascend 310P ~ 22-44 TOPS INT8 足够跑 BGE-M3 (1024 维 embedding), 1 张 NPU 即可承担 100 GB/s 数据流的 embedding 生成.

变现: 卖 Pre-vectorized Storage / AI-ready data lake, 跟 NVIDIA AIDP 同赛道但用 NPU 而非 GPU, 成本/能耗优势.

🥇 DIFF-2: 训练+推理统一存储 (Converged Training+Inference)

可行性: 高 vs WekaFS / VAST (走 NFS/S3) 类似 DDN A3I 但用 NPU + 本地 SSD

用户做什么: 256 节点刚好在 frontier model 训练边界 (256 GPU = 70B 模型 1 epoch 几天). checkpoint 落本地 SSD 时, 立刻被同一集群的推理节点直接 in-node 加载 (NPU 拉本地 SSD = PCIe 直连, 无网络).

变现: 卖 Converged Training+Inference 一体机.

🥈 DIFF-3: NPU 加速的数据集去重 / 质量评分

可行性: 中 vs NeMo Curator (NVIDIA GPU) vs VAST DataEngine (CPU)

用户做什么: 数据准备阶段, 用 NPU 跑语义去重 (semantic dedup, embedding 余弦相似度). 业界 NeMo Curator 用 GPU 占训练算力, 用户的 NPU 更便宜且节点内即可, 不占 GPU.

风险: NPU 软件栈 (CANN) 对 embedding 模型支持还在演进.

🥈 DIFF-4: 256 节点内的 Agent Sandbox 共享文件系统

可行性: 中-高 vs Modal / Anyscale (冷启秒级)

用户做什么: Modal / Anyscale 走 object store, agent 冷启秒级. 256 节点用户可以让 agent sandbox 落本地 SSD + RDMA 共享 (NPU 之间走 HCCL, 天然共享文件系统层). Cold start < 100ms.

关键: namespace 隔离 (per-tenant sandbox root), 类似 VAST ElementStore 的 B-tree KV.

DIFF-5: 256 节点级别的 RLHF / RL Rollout 共享 sample 缓冲

可行性: 中 vs Modal / Anyscale (对象存储)

用户做什么: RLHF 大量并发 sandbox 写小文件 + 读 sample. 256 节点 + 节点内 SSD 池 = 一个 256xNVMe 的统一 namespace, RL trainer 拉 sample 不走外网.

DIFF-6: 数据血缘 + 训练 lineage 强 hook (in-band provenance capture)

可行性: 中-高 vs OpenLineage (out-of-band) EU AI Act 合规卖点

用户做什么: NPU+SSD 紧耦合层做 IO 拦截 (类似 NVMe-oF ZNS), 每一个 read/write 自动带 provenance metadata (dataset hash, version, tenant). 业界 OpenLineage 是 out-of-band 上报 (不可靠), 用户的 in-band 抓取更可信.

护城河: 深. 改 NVMe-oF 控制器/客户端驱动, 别人难抄.

DIFF-7: 训练数据湖的零拷贝 snapshot + fork training job

可行性: 中 vs LakeFS (metadata 层) vs DVC (Git pointer)

用户做什么: LakeFS 是 metadata 层做 snapshot, 用户可以在本地 SSD 层做 (CoW snapshot + NPU 直接从 fork 读, 0 网络跳). 适合反事实实验 (数据换 v2/加 10% / 换特征工程).

风险: ZFS/BTRFS 在 256 节点 x TB 级 SSD 上要小心 fragmentation.

5.3 不该做的方向 (避免撞 Mooncake/LMCache/InfiniStore)

🔴 明确不做
  • 跨节点 KV 集群: Mooncake 千节点, LMCache vLLM 集成, InfiniStore ByteDance 出品. 已饱和, 软件壁垒高.
  • 通用对象存储: AWS S3 / MinIO / Cloudian 已是红海.
  • 通用并行文件系统: Lustre / GPFS / BeeGFS 已有成熟方案.

6下一步建议 — 我推荐先做这 3 个 PoC

🟢 3 个推荐 PoC (按 ROI 排序)
  1. PoC-1 (D1): NPU embedding-on-write 验证 — 选 1 个 embedding 模型 (BGE-M3 568M), 数据落 SSD 时 NPU 跑向量化, 测吞吐 / 延迟 / 准确度. 1-2 周, 单节点可验证.
  2. PoC-2 (D2): 训练+推理统一存储 — 256 节点训练 70B 模型, checkpoint 落本地 SSD, 推理节点同集群拉权重, 测 vs 走 NFS 的延迟差. 4-8 周, 需真实模型.
  3. PoC-3 (D6): in-band 数据血缘 — 改 NVMe-oF 控制器加 provenance metadata, 跑 EU AI Act 合规场景. 8-12 周, 投入大但护城河深.

6.1 立即可做的"反例验证" (1 周内)

在动手前, 我建议先做 3 个反例验证 排除错路:

  1. 跑 EAGLE draft 验证 8-12 张 310P 利用率 < 5% (从上一轮报告的 NPU 过配置结论). 测一下, 别光信报告.
  2. 测 310P KV 压缩 (KVQuant-style) 性能 看 NPU 算力够不够.
  3. 测 2 节点 GPU↔GPU RDMA 真实延迟 看"网络打 SSD"在用户的硬件上是真是伪命题.

6.2 商业模式建议

  • 不做: 卖纯 KV 集群软件 (已有 Mooncake/LMCache/InfiniStore)
  • 做: 卖 Converged Storage + NPU 一体机 (类似 DDN A3I 模式), 硬件绑定, 客户为"AI-ready"付溢价
  • 目标客户:
    • 中国主权云 (China Mobile/Telecom/Unicom)
    • 省级 AI 云 (政务云 / 国资云)
    • 私有部署的银行 / 制造 / 能源
    • 东南亚 / 中东主权云 (出口)
